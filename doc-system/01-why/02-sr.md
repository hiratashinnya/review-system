# ステークホルダー要求

> **型**: SR ／ **必須上流**: VAL（refines ✅）

---

## 文書を提出して指摘レポートを受け取りたい

<details><summary>⬡ SR-1 · v0.1</summary>

```yaml
id: SR-1
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-1
    kind: refines
    status: pending
    ref_version: "0.1"
```
</details>

利用者（書き手）が文書を提出すると、観点に沿った指摘が 🤖/✋/💬/❓ で仕分けられたレポートを受け取れる。

---

## 指摘への判断を反映して修正を適用したい

<details><summary>⬡ SR-2 · v0.1</summary>

```yaml
id: SR-2
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-1
    kind: refines
    status: pending
    ref_version: "0.1"
  - to: VAL-2
    kind: refines
    status: pending
    ref_version: "0.1"
```
</details>

レビュアーが承認・決定した修正が文書に反映でき、自動適用済みの修正はいつでも revert できる。

---

## 評価基準・ポリシーを組織単位で管理・改善したい

<details><summary>⬡ SR-3 · v0.1</summary>

```yaml
id: SR-3
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-3
    kind: refines
    status: pending
    ref_version: "0.1"
```
</details>

基準メンテナが評価基準ファイルとポリシーを外部ファイルで管理し、現場の却下傾向を受けてフィードバックを基準に還流できる。

---

## 自動化レベルをコンフィグで制御したい

<details><summary>⬡ SR-4 · v0.1</summary>

```yaml
id: SR-4
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-2
    kind: refines
    status: pending
    ref_version: "0.1"
```
</details>

組織が自動化ポリシー（`determinism × severity → 適用モード`）をファイルで設定し、自動修正の範囲を安全に制御できる。
