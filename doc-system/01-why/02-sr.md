# ステークホルダー要求

> **型**: SR ／ **必須上流**: VAL（refines ✅）

---

## SR-1: 著者が型・ID・辺ルールを間違えずに書ける

<details><summary>⬡ SR-1 · v0.1.0</summary>

```yaml
id: SR-1
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-3
    ref_version: "0.1.0"
```
</details>

仕様著者が、型・ID・辺のルールを間違えずにノードを書けることを求める。

---

## SR-2: レビュアーが機械的に網羅性・整合性を検証できる

<details><summary>⬡ SR-2 · v0.1.0</summary>

```yaml

type: SR

scheduled: ""

  - to: VAL-2
    ref_version: "0.1.0"
```
</details>

---

## SR-3: メンテナが変更の影響範囲を辺でたどれる

<details><summary>⬡ SR-3 · v0.1.0</summary>

```yaml

type: SR

  - to: VAL-1
```

---

> ステークホルダーの欲求ではなく **SR-2（機械検証）を系の視点で言い換えた重複**であり、形式制約は NFR-1/2/6 が表現済み。
> 旧 SR-4 の被依存（FR-1/FR-9/NFR-1/NFR-2/NFR-6）は SR-2 へ再配線した（消さない＝経緯は DD-1 に記録）。

## SR-5: 後工程の修正・指摘から影響する上流ノードを特定できる

<details><summary>⬡ SR-5 · v0.1.0</summary>

```yaml
id: SR-5
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-1
    ref_version: "0.1.0"
```
</details>

レビュアー・テスト担当者が、後工程で発見した指摘・失敗から、影響する上流ノード（FR/SR/VAL）を辺をたどって特定できることを求める。

---

## SR-6: 上流変更の下流への反映漏れ（ドリフト）を継続検出できる

<details><summary>⬡ SR-6 · v0.1.0</summary>

```yaml
id: SR-6
type: SR
labels: []
scheduled: ""
edges:
  - to: VAL-2
    ref_version: "0.1.0"
```
</details>

メンテナ・レビュアーが、上流の仕様変更・意思決定が下流に未反映なまま放置されるドリフト（腐敗源）を、機械的に継続検出できることを求める。
