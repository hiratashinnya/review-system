# 機能要求

> **型**: FR ／ **必須上流**: SR（refines ✅）
> doc-system＝Markdown 埋め込みの型付き ID グラフ＋ドリフト/構造/カバレッジを検証するツール。
> 出典：`docs/doc-system/`（DD-001〜011・RULE-001〜021・接続マトリクス・config.yaml）。

`PREFIX-N[-N...]` の連番・永続 ID を採番し、階層分割（`I-1`→`I-1-1`）は **ID パターンから推論**する（`decomposes` 辺は廃止・DD-014）。ID は永続（リネームしても変えない、意味は heading が持つ）。上下双方向のグラフ走査を可能にする（DD-002/003・02§3）。

## FR-8: 三軸の検査抑制機構

<details><summary>⬡ FR-8 · v0.2</summary>

```yaml
id: FR-8
type: FR
labels: []
scheduled: ""
edges:
  - to: SR-7
    ref_version: "0.2"
```
</details>

検査ノイズを 3 つの軸で抑制する：①`scheduled`（後フェーズ＝完全サイレント）、②ステージ発火（config の `activate_stage`/`rule_activation` が current_stage 未達なら沈黙・旧 stage_scope.disable を代替）、③`suppress`（ルール単位・理由必須・ノード単位サイレント）。`always_error`（RULE-005/007）はいずれでも抑制不可（DD-007/019・02§8）。

## FR-9: トレース対象集合の宣言

<details><summary>⬡ FR-9 · v0.3</summary>

```yaml
id: FR-9
type: FR
labels: []
scheduled: ""
suppress: [RULE-018]  # 異常系なし: 走査範囲は静的な include/exclude 宣言で fail パスを持たない
edges:
  - to: SR-2
    ref_version: "0.2"
  - to: DD-1
    ref_version: "0.1"
```
</details>

`config.yaml` の `trace_scope.include`/`exclude`（glob）で検証対象（in-graph）ファイル集合を宣言し、doc-system 自身・methods・README 等を out-of-graph として除外する（exclude が include より優先）（06）。

## FR-10: 検証ツールの実行形態

<details><summary>⬡ FR-10 · v0.2</summary>

```yaml
id: FR-10
type: FR
labels: []
scheduled: ""
suppress: [RULE-018]  # 異常系なし: 出力契約（終了コード・整列）は正常系のみ。検出された違反は ERROR ではなく出力内容
edges:
  - to: SR-2
    ref_version: "0.2"
```
</details>

検証は Python CLI（標準ライブラリのみ）として実行し、ERROR があれば終了コード 1・なければ 0 を返す。出力は深刻度順（ERROR→WARNING→INFO）に整列する。段階①②を先行実装、③を次、④は sub-issue（05§6）。
