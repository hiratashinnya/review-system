# 機能要求

> **型**: FR ／ **必須上流**: SR（refines ✅）
> doc-system＝Markdown 埋め込みの型付き ID グラフ＋ドリフト/構造/カバレッジを検証するツール。
> 出典：`docs/doc-system/`（DD-001〜011・RULE-001〜021・接続マトリクス・config.yaml）。

`<details><summary>⬡ PREFIX-N` トグルと YAML ブロックで、型・ID・labels・辺を持つノードを Markdown に埋め込み、検証ツールがパースできる構造化表現を提供する（DD-001・04 記法）。

## FR-2: ID 体系と階層分解

<details><summary>⬡ FR-2 · v0.2</summary>

```yaml

  - to: [SR-3, SR-5]
```

`PREFIX-N[-N...]` の連番・永続 ID を採番し、階層分割（`I-1`→`I-1-1`）を親の `decomposes` 辺で表現する。ID は永続（リネームしても変えない、意味は heading が持つ）。上下双方向のグラフ走査を可能にする（DD-002/003・02§3）。

## FR-8: 三軸の検査抑制機構

<details><summary>⬡ FR-8 · v0.2</summary>

```yaml
id: FR-8
type: FR
labels: []
scheduled: ""
edges:
  - to: SR-7
    kind: refines
    status: pending
    ref_version: "0.2"
```
</details>

検査ノイズを 3 つの軸で抑制する：①`scheduled`（後フェーズ＝完全サイレント）、②`stage_scope`（`disable: [{rule, on: NodeType}]` ペアのリストでステージ単位に特定ルール×ノードタイプの検査を完全サイレント）、③`suppress`（ルール単位・理由必須・ノード単位サイレント）。`always_error`（RULE-007）はいずれでも抑制不可（DD-007・02§8）。

## FR-10: 検証ツールの実行形態

<details><summary>⬡ FR-10 · v0.2</summary>

```yaml
id: FR-10
type: FR
labels: []
scheduled: ""
suppress: [RULE-018]  # 異常系なし: 出力契約（終了コード・整列）は正常系のみ。検出された違反は ERROR ではなく出力内容
edges:
  - to: SR-2
    kind: refines
    status: pending
    ref_version: "0.2"
```
</details>

検証は Python CLI（標準ライブラリのみ）として実行し、ERROR があれば終了コード 1・なければ 0 を返す。出力は深刻度順（ERROR→WARNING→INFO）に整列する。段階①②を先行実装、③を次、④は sub-issue（05§6）。
