## PEND-1: I-1-1/I-1-2/I-1-3 の独立入力化の妥当性（PR1 過分割の疑い）

**status: resolved**（2026-06-13 初回解消＝I-1-1/I-1-2/I-1-3 を I-1 の子ノードとして改名・親辺 `to: I-1` を付与して階層を明示。2026-06-15 DD-12 退役で再解消＝3 ノードを退役し D-18/D-19 の派生属性へ吸収。独立入力扱いの問題を根本是正。）

<details><summary>⬡ PEND-1 · v0.2.0</summary>

```yaml
id: PEND-1
type: PEND
labels: []
scheduled: ""
edges:
  - to: D-18
    ref_version: "0.1"
  - to: FND-6
    ref_version: "0.2"
  - to: DD-4
    ref_version: "0.1"
```
</details>

**論点**: I-1-1（suppress 設定）・I-1-2（scheduled 設定）・I-1-3（ref_version 値）を I-1（ノードファイル）と独立した入力ノードとして割ることが妥当か。いずれも仕様著者がノードファイル内に記述する値であり、発生源と「もの（ファイル）」が同一である。PR1（もの＋発生源で分ける／使い道だけで割らない）に照らすと過分割の疑いがある。

**先送り理由**: 要件段では I-1-1/I-1-2/I-1-3 がそれぞれ別の SPEC（SPEC-22/SPEC-20/SPEC-9）に紐づいており、追跡価値が分割コストを上回る。統合に伴う SPEC 側の再編は影響範囲が大きく、現フェーズでの統合コストが過大と判断。FND-6（INFO）として記録済みであり、緊急度は低い。

**再検討条件**: 設計段（DM/CFG 設計時）で I-1-1/I-1-2/I-1-3 の型設計を行う際に再評価する。統合案（I-1 に吸収）と現状維持案のトレードオフを DM 設計文脈で比較する。

**解消経緯①（2026-06-13）**: I-1-1/I-1-2/I-1-3 を I-1 の子ノードとして改名し階層を明示（独立入力扱いの緩和）。FND-6 と連動して resolved。

**解消経緯②（2026-06-15・DD-12 退役による supersede）**: 分析層全面見直し（DD-12）で I-1-1/I-1-2/I-1-3 を退役（系外入力でなく D-4 派生属性＝PR1/PR9 是正）。suppress/scheduled は D-18（属性検査ビュー）へ、ref_version は D-19（決定辺ビュー）へ吸収。本 PEND の義務辺は退役で消える `to: I-1-1` から、suppress の吸収先 **`to: D-18` へ repoint** した。

**DD-4 追跡の moot 化注記**: DD-4（分析層 ref_version ドリフト群の一括解消・2026-06-13）は「生きた義務辺 PEND-1→I-1-1」を更新対象として扱い、ref_version を "0.5"→"0.6" に追跡してきた（decisions.md DD-4 ドリフト群表）。DD-12 の退役により **I-1-1 そのものが消滅したため、DD-4 が追跡していた PEND-1→I-1-1 の ref_version ドリフト追跡は moot（無効化）となる**。本改訂で当該義務辺を D-18 へ repoint したことで、以後の ref_version 追跡対象は D-18 のバッジ（現 v0.1）に切り替わる。DD-4 自体は決定済み凍結記録として保持し（PR8・消さない）、`to: DD-4` 辺は経緯参照として維持する。

**関連**: FND-6（INFO・過分割の疑いを記録した指摘ノード・DD-12 で supersede 解消）／DD-4（PEND-1→I-1-1 義務辺の旧 ref_version 追跡・退役で moot）／DD-12（I-1-x 退役を決定した上位決定）

## PEND-2: 分析層の図（コンテキスト図・DFD）の手動メンテをスクリプト自動生成へ置換（先送り）

**status: deferred**（sprint-2 以降・VAL-5/FR-15 で実装）

```yaml

edges:
  - to: VAL-5
    ref_version: "0.1"
  - to: FR-15
    ref_version: "0.1"
```
</details>

**論点**: 現状コンテキスト図・DFD（00-dfd.md）はノードグラフから手動で描き起こしており、ノード変更のたびに手動同期が必要でドリフトの温床になる。最終的にはノードグラフからスクリプトで各図を機械生成し、手動メンテを廃したい。
**先送り理由**: 図の機械生成は VAL-5（グラフの図的可視化・SR-8・sprint-2 scheduled）と FR-15（--export-graph dot/json）で既に価値命題・機構として捕捉済み。本 PEND はそれらの実装フェーズ（sprint-2 以降）に統合して扱う。要件・分析フェーズでは手動図を維持し、図とノードの並走（DD-7 ⑤）でドリフトを抑える。
**再検討条件**: sprint-2 で VAL-5/FR-15 の実装に着手する際、コンテキスト図・DFD のスクリプト生成を具体化する。
**関連**: VAL-5（図的可視化）・FR-15（依存グラフレポート出力）
