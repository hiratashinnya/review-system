</details>

**status: decided**（2026-06-13 反映完了）

**論点**: VERIFY ノードや解消済み FND など「ある時点の状態を記録した凍結記録」の `ref_version` は、参照先ファイルが版上げされるたびに RULE-004 ドリフト WARNING を発火し続ける。これらを (A) 凍結スナップショットとして RULE-004 免除する、(B) 都度 ref_version を最新化する、(C) ドリフトを「再検証シグナル」として据え置く、のいずれの意味論を採るか。（Q-1 より昇格）

**選択肢**:
- **A（免除）**: VERIFY・resolved FND に `suppress: [RULE-004]` を付与し、凍結スナップショットとして RULE-004 を免除。ノイズ最小、ただし「いつの版を見たか」を本文に残すことが前提。
- **B（都度更新）**: 記録系も生きた依存辺として ref_version を最新化。一貫するが、実際にはレビューしていない版を指す矛盾が生じうる。
- **C（シグナル据え置き）**: ドリフト発火を再検証シグナルとして活用し、陳腐化したら新 VERIFY/FND を起票。監査性は高いが恒常的なドリフト WARNING が残る。

**決定**:
- **VERIFY ノード**: A を採用。`suppress: [RULE-004]` を付与し「過去の検証事実スナップショット」として凍結。理由：VERIFY は「いつ・何の版を・どのようにレビューしたか」という過去事実の記録であり、参照先の将来変更で書き換えると証跡として不正確になる。本文に実施日・対象版を記載することを前提条件とする。
- **FND（解消済み含む）**: C を採用。ドリフト発火を「記録が陳腐化 → 再検証を検討」のシグナルとして活用する。処置後に辺が逆転（target → FND）してもドリフト発火は有効な再検証シグナルとなる。

**影響範囲**:
- `04-verification/01-doc-verify.md`: VERIFY-1・VERIFY-2 に `suppress: [RULE-004]` を付与（v0.1.1→0.1.2・z バンプ）。各ノードに `→ DD-2` バックリファレンス辺を付与。
- `04-verification/05-questions.md`: Q-1 を `status: closed`（DD-2 へ昇格）に更新（v0.1.0→0.1.1）。
- グラフ外プロセス文書（config.yaml 等）: 型別 RULE-004 免除の機構設計は設計フェーズで行う。

## DD-3

<details><summary>⬡ DD-3 · v0.1</summary>

```yaml

# 反映済み。影響範囲はすべてグラフ外プロセスドキュメントのため
# in-graph 義務辺は不要。
```
</details>

**status: decided**（2026-06-13 反映完了）

**論点**: FND 解消時に edges が「FND → 対象」から「対象 → FND」へ逆転するため、元の指摘時に記録されていた `ref_version`（どの版の対象を指摘したか）が辺情報から失われる。解消後の FND は辺の方向が反転してしまうため、指摘当時の参照版を追跡できなくなる問題をどう解消するか。

**選択肢**:
- **A（本文明記ルール）**: FND を起票する際、`edges[].ref_version` の値を本文にも明記するルールを制度化。「**指摘時 ref_version**: SPEC-1 "0.3"（spec.md v0.3.1 時点）」のような形式で記録する。— 実装コストゼロ・プロセス追加のみ。辺が逆転しても本文が証跡を保持。
- **B（専用属性追加）**: FND ノードに `target_ref_version` などの専用メタ属性を追加。— 機械可読性は高いが、スキーマ拡張コスト・既存 FND の遡及修正コストが大きい。
- **C（バックリファレンス辺の ref_version を使う）**: バックリファレンス辺（target → FND）の ref_version に「処置時の版」を記録する。— 辺モデルの意味を「指摘時の版」から「処置時の版」に変える副作用があり混乱を招く。

**決定**: A を採用。FND 起票時に `edges[].ref_version` の値を本文に明記することをルールとして制度化する。理由：実装コストゼロ・既存 FND への遡及対応が可能・バックリファレンス辺の意味論（処置時の辺）を汚染しない。B はスキーマ拡張を要し、設計フェーズより先に規約コストを作ることになる。

本文への記録形式（標準テンプレート）:
```
**指摘時 ref_version**: {対象ノードID} "{ref_version 値}"（{ファイル名} v{version} 時点）
```
例:
```
**指摘時 ref_version**: SPEC-1 "0.3"（spec.md v0.3.1 時点）
```

**影響範囲**（いずれもグラフ外プロセスドキュメント・in-graph 義務辺は不要）:
- `.claude/agents/verification-author.md`: FND 著作ルールに「`edges[].ref_version` を本文にも明記する」を追加。
- `docs/doc-system/07-authoring-guide.md`: FND 解消ライフサイクルセクションに上記ルールと記録形式テンプレートを追記。
- `.claude/skills/spec-principles/SKILL.md`（または該当箇所）: PR7「FND を resolved にしたら処置対象に `→FND-x` 辺を付与」の注記として、「起票時に ref_version を本文へ明記することで解消後も指摘時の版が追跡可能になる」を追加。
- `CLAUDE.md`「FND 解消ライフサイクルとバックリファレンス辺」セクション: 同様の注記を追加。
