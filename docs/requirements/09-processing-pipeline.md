# 09. 処理パイプライン（intake → レポート）

これまで部品ごとに設計した intake([08](08-intake-design.md)) / 基準([01](01-criteria-files.md)/[schema](../schema/README.md)) /
AI 入出力([07](07-ai-input-design.md)) / 仕分け・提示([02](02-evaluation-and-triage.md)) / 自動修正([03](03-auto-fix-policy.md)) を
**1本の流れに通し**、順序の不変条件と継ぎ目の穴を洗い出す。

## パイプライン

```
①intake ─▶ ②基準合成 ─▶ ③プロンプト構築 ─▶ ④LLM評価 ─▶ ⑤正規化&検証 ─▶ ⑥仕分け ─▶ ⑦自動適用 ─▶ ⑧レポート ─▶ ⑨フィードバック
```

| 段 | 入力 | 出力 | 関連 | 注意・不変条件 |
|---|---|---|---|---|
| ① intake | 関連文書群 / 参照 / 型 / スコープ | **評価対象集合**・**参照集合**（ファイル所属）/ doc_type / scope=org | 08 | 後段の参照除外はこの集合判定に依存 |
| ② 基準合成 | doc_type × scope | 観点パック（id/title/本文/例）＋ **メタ表**(determinism/severity/override・provenance) | 01/schema | MVP は org 単一。兄弟ファイル union＋本文矛盾チェック(Q10)はここ |
| ③ プロンプト構築 | 観点パック / 対象 / 参照 | LLM 入力 | 07 | **メタは載せない**（仕分けは後段）。対象と参照を分けて渡す |
| ④ LLM 評価 | プロンプト | findings[] / unmatched[]（**location.file 必須**） | 07 | LLM は発見＋id付与＋原案のみ。仕分けしない |
| ⑤ 正規化&検証 | ④の出力＋①の集合 | クリーンな findings / unmatched | 07 | ★**仕分け・適用より前**。下記2処理 |
| ⑥ 仕分け | findings＋メタ表＋ポリシー | 🤖/✋/💬 ・❓未分類 | 02/03 | `rule_id → determinism×severity → ポリシー → モード` |
| ⑦ 自動適用 | 🤖 の修正案 | 評価対象への書込＋**指摘(finding)単位のコミット** | 03 | **評価対象集合のみ書込**。finding ごとに1コミット（revert 粒度＝finding） |
| ⑧ レポート | ⑥⑦の結果 | 3区分＋未分類・サマリ・diff・原案 | 02 | O-1..O-5・O-7 |
| ⑨ フィードバック | 人間の操作 | 育成（O-12 等） | 04 | I-6/I-7 → O-12 |

### ⑤ 正規化&検証（継ぎ目の要）

仕分け・適用に進む前に、プログラムが2つを必ず行う：

1. **rule_id 検証** — finding の `rule_id` が②で送った観点パックに存在するか確認。
   無い（LLM が実在しない id を付けた）ものは **❓未分類へ回す**（Q7 の対象は「LLM 自己申告の unmatched」だけでなく「検証で外れた id」も含む）。
2. **参照除外** — `location.file` が**参照集合**に属する finding を落とす（07）。LLM の行儀に依存しない機械除外。

## 順序の不変条件（崩すと事故）

- **参照除外・rule_id 検証（⑤）は、仕分け（⑥）・自動適用（⑦）より必ず前。** さもないと参照ファイルを書き換える/偽 id を仕分ける。
- **自動適用（⑦）が書き込むのは評価対象集合のファイルだけ。** 参照・未分類・✋・💬 は絶対に書き換えない。
- **メタ（severity/determinism/override）はプロンプト（③）に出さない**（07 原則・2軸）。

## 通して見えた論点

- **Q20【新規】 同一実行内の自動修正衝突** — 🤖 が同じファイル/行に重なる修正を出した場合の適用順序・衝突解決が未定義。⑦の前提。
- **Q3 に既定が出た** — revert の単位は **指摘(finding)単位でコミット**（finding id＝rule_id＋location）。
  個別 revert も、「この実行ぶん全部」も、対象コミット群をまとめて revert すれば成立（一括は単位でなく操作）。all-or-nothing にしない。
- **Q14 はクローズ** — パイプラインが1実行ごとにステートレスなのは**正しい設計**。再提出は単に再レビュー（重複指摘が出てよい／指摘は問題が残る限り毎回出すべき）。レビュー指摘に dedup 状態は持たせない。※"警告"の dedup は別物で警告レジャー（Q9/schema）が担う。
- **Q17 リンク** — ②〜④はどこでも失敗し得る（合成失敗・LLM障害・パース失敗）。失敗時は部分結果＋エラー通知(O-14)へ degrade する設計が要る。
