# 03. 自動修正ポリシー

## 設計思想

自動化の度合いを **組織がコンフィグで決められる** ようにする。
信頼が育つにつれて自動化レベルを上げていける ＝ ツールの寿命が延びる。

A×B マトリクスの各セルに「適用モード」を設定できる。

## 自動化レベルの目安

| 指摘の性質 | デフォルト | 根拠 |
|---|---|---|
| 決定論的に直せる（リントエラー、フォーマット、誤字、命名規則違反） | **Auto-fix（ログのみ・承認スキップ）** | 答えが一意。人間が見る価値が薄い |
| トレードオフはあるが定石が明確（軽微なリファクタ、冗長表現の整理） | **Auto-fix（diff 提示・承認任意）** ※承認必須にも設定可 | AI のトレードオフ分析で大崩れしない |
| 要件矛盾・欠落 | **Human-only（AI は指摘＋原案提示まで）** | 意思決定が絡む。適用は人間 |
| 仕様の検討・設計判断 | **Suggest（AI が原案・複数案、人間が選択）** | 原案生成は AI で十分価値が出る |

## ポリシーの階層（継承）

基準と同じく `全社デフォルト → チーム → プロジェクト` で上書き可能。ただし**方向ゲート制（org 権威）**で縛る（→ [01](01-criteria-files.md) / [schema](../schema/README.md)）。

- **自動化を減らす方向**（人を増やす。「定石リファクタも承認必須にする」等）→ **常に自由**。
- **自動化を増やす方向**（`human_only`→`auto_fix_suggest`→`auto_fix_log_only`。人の関与が減る）→ **既定で機械拒否**。org が `open` にしたルールのみ可。`override: locked` は変更不可。

## 安全装置

1. **自動修正サマリで必ず可視化** — 「ログのみ」で適用したものも、
   評価レポートに「🤖 自動修正済み」として必ず添える（サイレントに直さない）。
   デフォルト折りたたみで流し読みの邪魔をしない。
2. **revert** — 自動適用を **指摘(finding)単位でコミット**して残す（内部ローカル git・Q3）。
   個別の指摘だけ戻すことも、実行ぶんをまとめて戻すこともできる（一括は単位でなく操作）。
   粒度が細かいほど組織は安心して自動化レベルを上げられる。
3. **diff 承認** — auto-fix（承認任意以上）は「提案 → 承認 → 適用」を挟む。
   いきなり書き換えると信頼を失う。

## 決定済みの線引き

- **修正案の生成主体も境界で分ける（Q21）**：整形/import整列/lint自動修正/誤字/ケース変換など**決定的ツールがあるもの**はツールが生成し、判断を含むものだけ LLM が生成（[10](10-llm-system-boundary.md)）。
- 決定論的なもの（ルール違反・エラー修正）→ **ログのみ** で良い
- AI のトレードオフ分析でそうそう外さないレベル → **コンフィグ次第で auto-fix 対象に**
- 要件矛盾・欠落 → **人間の確認を入れる**（ただし原案検討は AI が行う）
- 自動修正サマリは **人間が流し読みする運用**（記録に留めず視界に入れる）
