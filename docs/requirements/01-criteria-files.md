# 01. 評価基準ファイル

## 方針

評価基準を **構造化 Markdown（メタ情報付き）** として外部ファイル化し、
Git でバージョン管理する。

### メリット

- バージョン管理しやすい（履歴・差分・レビューが回る）
- ノンエンジニアでも編集可能
- 「基準を育てる」仕組みと相性が良い（PR ベースで承認フローを作れる）

## 構造化の方針

完全な自由記述だと AI の評価がブレるため、ある程度スキーマを決める。
各観点が最低限持つ項目（案）:

- 観点名
- 重要度
- 深刻度（Error / Warning / Info）
- 対応モード（auto-fix / suggest / human-only）
- determinism（決定論的に直せるか）
- 良い例 / 悪い例
- チェック方法

## 動的選択のキー

基準は二次元で選択する:

```
適用基準 = 文書タイプ × スコープ（全社 / チーム / プロジェクト）
```

文書タイプだけでなく、プロジェクト・チーム軸でも切り替えられるようにする。

## 基準の合成（継承）

「全社共通基準 + チーム固有基準」を重ねられるよう、
**継承 / オーバーライド** の仕様を最初に決めておく。

```
全社デフォルト  →  チーム  →  プロジェクト
（下位が上位を上書きできる）
```

この継承の仕組みは、自動化ポリシー（03）でもそのまま流用する。

## スキーマ（v0 叩き台）→ [../schema/README.md](../schema/README.md)

A1 で実際に書いた。要点：

- **二層構成**：YAML フロントマター（機械可読・ルーティング層）＋ Markdown 本文（観点＝人間 & LLM 共用）。
  情報の読み手（人間 / プログラム / LLM）で欲しい形が違うため層を分けた。
- **`id` が全ての結合キー**（指摘の紐付け・継承マージ・ポリシー写像）。
- **対応モード（auto-fix 等）は基準ファイルに書かない**。基準はルールの素性（`severity` / `determinism`）だけ宣言し、
  適用モードへの写像は別の**ポリシーファイル**が担う（責務分離）。
- **LLM は仕分けしない**。違反を見つけ `id` を付け原案を出すだけ。仕分けはプログラムが
  `id → determinism × severity → ポリシー` で機械的に行う。

```yaml
# フロントマター（抜粋）
rules:
  - id: naming-convention
    title: 命名規則
    severity: error
    determinism: deterministic   # auto-fix 等の適用モードはここに書かない
    enabled: true
```
