# 08. 入力の受け取り設計（intake）

レビュー対象がシステムに入る部分の設計。MVP の方針を確定する。
扱うのは **評価対象の単位 / 参照コンテキスト / 文書タイプ判定 / スコープ決定** の4点。

## 評価対象の単位（Q12）

**1レビュー = 提出された関連文書群（1〜N ファイル）をまとめて評価**。

- 単一文書（議事録・仕様書）はそのまま1ファイル。コード変更のように複数ファイルが関連する場合は**まとめて1レビュー**にし、ファイル横断の整合（例：呼び出し側と定義側）も見られるようにする。
- アップロード（ファイル/テキスト）が基本。PR diff は一形態にすぎず Git 前提にはしない。

## 参照コンテキスト（上流成果物・前提条件）＝ I-13【新規】

**評価対象とは別枠で、突き合わせ用の上流情報を入力できる。** これが従来の台帳に欠けていた入力。

| 評価対象 | 典型的な参照コンテキスト |
|---|---|
| コード | 仕様書 / 設計書 / チケット |
| 仕様書 | 要件定義 / 上位仕様 / 制約条件 |
| 議事録 | アジェンダ / 過去の決定事項 |

設計上の扱い：
- 評価対象と区別して渡すが、**参照側を指摘しないことを LLM に厳密強制しない**。代わりに
  **レポート生成時に location で機械的に除外**する（参照ファイル由来の finding を落とす）。intake が
  「評価対象集合／参照集合」のファイル所属を保持し、後段がそれで判定する（[07](07-ai-input-design.md)）。
- **網羅性・整合性・前提逸脱**といった観点は、この参照が無いと判定できない。参照が無ければそれらの観点はスキップ/低信頼として扱う。
- MVP は**手動添付・任意・複数可**。どの観点がどの参照を要求するか等の紐付け細部は [Q19](../dashboard.md)。

## 文書タイプ判定（Q4）

**AI 自動判定を既定、ユーザーが手動上書き可。** 確信度が低いときは手動選択を促す。
タイプ（`code`/`spec`/`minutes`/…）は適用基準（`doc_type`）を選ぶルーティングキー（[01](01-criteria-files.md)）。
誤判定すると誤った基準が適用されるため、上書きを常に効かせる。

## スコープ決定（Q13）

**MVP は org（全社デフォルト）単一スコープ。** `team`/`project` の継承・union 機構は設計済み（[schema](../schema/README.md)）だが MVP では露出しない。
スコープ選択 UI と紐付け（明示選択）は MVP 後に追加する。アクター区別を持たない MVP 方針とも整合（誰の文書か≠スコープ）。

## 入力まとめ（台帳との対応）

```
[評価対象の関連文書群]  … I-1（単位=バッチ, Q12）
[参照コンテキスト]      … I-13（任意・突き合わせ用）【新規】
[文書タイプ]           … I-2（AI判定＋手動上書き, Q4）
[スコープ]             … I-3（MVPは org 固定, Q13）
   ↓
合成基準（I-4, MVPは org のみ）＋ ポリシー（I-5）で評価 → レポート
```
