# 11. プラットフォーム・アダプタ方針

## 前提（方針）

- **LLM は自前実装しない。** Claude Code / GitHub Copilot 等の**プラットフォーム（PF）に任せ**、本システムは
  **PF のラッパー／オーケストレータ**として作る。
- PF との接続部は**アダプタパターン**で抽象化し、PF を差し替え・追加できる拡張性を保つ。
- いざとなれば**自前実装（`SelfHostedAdapter`）**も同じポートの後ろに差せる（フォールバック）。

## どこが PF か（[10](10-llm-system-boundary.md) との接続）

doc10 で洗い出した LLM 呼び出し **6箇所（L1〜L6）はすべてアダプタ経由で PF** に投げる。
システム側の決定的処理（検証・参照除外・仕分け・適用・revert・方向ゲート）は**不変**。アダプタは「PF への問い合わせ」を正規化するだけ。

> doc10 の不変条件「全 LLM 出力はシステム検証か人フォールバックを通る」は、**アダプタが返す出力も同様に検証する**ことで保たれる。
> PF が何であれシステムは出力を信用しすぎない（rule_id 検証・参照除外・スキーマ検証はシステム側）。

## ポート（システムが依存する抽象）

システムは具体 PF ではなく**ポート（インターフェース）**に依存する。最小の契約：

- `review(観点パック, 評価対象, 参照, 出力スキーマ) → { findings[], unmatched[] }`（L1）
- 必要に応じ：矛盾判定(L2) / 型推定(L3) / 衝突マージ(L4) / 草案生成(L5,L6)
- いずれも「**構造化された問い → 構造化された答え**」。アダプタが PF 固有のやり方で実装する。

## アダプタ（具体）

- `ClaudeCodeAdapter` / `CopilotAdapter` / …（将来 `SelfHostedAdapter`）
- 各アダプタは**能力（capabilities）を宣言**する：構造化出力 / ファイル適用 / ツール実行 など。
- **対象の橋渡しもアダプタの仕事**：アップロード文書を PF の作業形態へ（例：一時作業ディレクトリに展開して Claude Code に渡す＝Q3 の内部ローカル git ワークスペースと同居）。
- **能力差はシステムが吸収**：
  | PF にできないこと | システムの代替 |
  |---|---|
  | ファイル適用 | 返された diff をシステムが適用（finding 単位コミット・Q3） |
  | ツール実行 | 決定論的修正（Q21）はシステムが formatter/linter を直接回す |
  | 構造化出力が弱い | アダプタがパース／再問い合わせで吸収（失敗は ❓未分類 / Q17 へ） |

## 論点

- **Q22【新規】 PF 能力差の吸収** — 能力宣言の項目と、フォールバックの線引き（最低限どの能力を必須にするか。
  例：**構造化出力は必須**／ファイル適用・ツール実行は任意でシステムが代替）。
- **Q5（技術スタック）** — AI レイヤは**外部 PF に委譲**で決着。残るはアプリ本体の実装言語・ホスティング。
