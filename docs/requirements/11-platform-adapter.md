# 11. プラットフォーム・アダプタ方針

## 前提（方針）

- **LLM は自前実装しない。** Claude Code / GitHub Copilot 等の**プラットフォーム（PF）に任せ**、本システムは
  **PF のラッパー／オーケストレータ**として作る。
- PF との接続部は**アダプタパターン**で抽象化し、PF を差し替え・追加できる拡張性を保つ。
- いざとなれば**自前実装（`SelfHostedAdapter`）**も同じポートの後ろに差せる（フォールバック）。

## どこが PF か（[10](10-llm-system-boundary.md) との接続）

doc10 で洗い出した LLM 呼び出し **6箇所（L1〜L6）はすべてアダプタ経由で PF** に投げる。
システム側の決定的処理（検証・参照除外・仕分け・適用・revert・方向ゲート）は**不変**。アダプタは「PF への問い合わせ」を正規化するだけ。

> doc10 の不変条件「全 LLM 出力はシステム検証か人フォールバックを通る」は、**アダプタが返す出力も同様に検証する**ことで保たれる。
> PF が何であれシステムは出力を信用しすぎない（rule_id 検証・参照除外・スキーマ検証はシステム側）。

## ポート（システムが依存する抽象）

システムは具体 PF ではなく**ポート（インターフェース）**に依存する。最小の契約：

- `review(観点パック, 評価対象, 参照, 出力スキーマ) → { findings[], unmatched[] }`（L1）
- 必要に応じ：矛盾判定(L2) / 型推定(L3) / 衝突マージ(L4) / 草案生成(L5,L6)
- いずれも「**構造化された問い → 構造化された答え**」。アダプタが PF 固有のやり方で実装する。

## アダプタ（具体）

- `ClaudeCodeAdapter` / `CopilotAdapter` / …（将来 `SelfHostedAdapter`）
- 各アダプタは**能力（capabilities）を宣言**する：構造化出力 / ファイル適用 / ツール実行 など。
- **対象の橋渡しもアダプタの仕事**：アップロード文書を PF の作業形態へ（例：一時作業ディレクトリに展開して Claude Code に渡す＝Q3 の内部ローカル git ワークスペースと同居）。
- **能力差はシステムが吸収**：
  | PF にできないこと | システムの代替 |
  |---|---|
  | ファイル適用 | 返された diff をシステムが適用（finding 単位コミット・Q3） |
  | ツール実行 | 決定論的修正（Q21）はシステムが formatter/linter を直接回す |
  | 構造化出力が弱い | アダプタがパース／再問い合わせで吸収（失敗は ❓未分類 / Q17 へ） |

## 双方向の口（PF ⇄ システム）

PF に LLM を委譲する構成なら、口を**双方向**にできる：

- **System → PF**（既述）：システムが PF に「レビューして」と問い合わせる（ヘッドレス運転向き）。
- **PF → System**（追加）：システムの**決定的オペレーションをツール（function calling / MCP サーバ）として公開**し、
  PF 側のエージェント（Claude Code 等）が呼ぶ。

同じ**ポート（契約）を両向きから使う**だけで土台は共通。公開するツール（＝これまで設計した決定的処理）：

- `compose_criteria(doc_type, scope)` → 観点パック＋メタ表（毎回合成・矛盾チェックは hash キャッシュ・Q15）
- `validate_findings(findings, pack)` → 有効／❓未分類 に仕分け（rule_id 検証・[09](09-processing-pipeline.md)⑤）
- `exclude_reference(findings, 参照集合)` → 参照由来を除外
- `triage(findings, メタ, ポリシー)` → 🤖/✋/💬
- `apply_fix` / `commit` / `revert`（finding 単位・内部 git・Q3）
- `run_deterministic_fix(file)` → formatter/linter（Q21）
- `warning_ledger_check(hash)` → 既出/未出（Q9）
- `build_prompt(...)`（プロンプト雛形）／`build_report(...)`

### MVP 戦略：システム＝ツール群、Claude が LLM 役

これを使うと **MVP を「決定的ツール群（MCP サーバ）＋プロンプト雛形」だけで立てられる**：

- 機械的な所とプロンプト周りをシステムとして実装する。
- **Claude Code 上で Claude が LLM 役**（観点違反の発見・id 付与・原案）を担い、上記ツールを呼んで
  合成→検証→仕分け→適用→レポートを回す。
- 専用オーケストレータも UI も**まだ作らない**。人間＋Claude が回しながら**プロンプトと機構を高速にブラッシュアップ**できる。
- ここで作るツールは捨てない：後で「System → PF」のヘッドレス運転に切り替えるとき、**同じ関数をシステム側オーケストレータが呼ぶ**だけ。

### PF 駆動の実装（CLI ＋ stdout で制御フローを流す）

PF 駆動は **CLI 前提**（Claude Code）。MCP/function-calling を組まずとも、**システムが制御フローを標準出力に流し**、
エージェント（Claude）はそれに従って各ステップ（LLM 役の推論・次コマンド実行）をこなす形でよい。

- **起動プロンプトは最小**：「**起動方法（CLI コマンド）＋入力（パス等）＋あとはシステムの指示（stdout）に従え**」。
- システムが「次はこれをしろ」を stdout で順に指示する＝**順序はシステムが握る**（純プロンプト依存より強い）。
- ツールは MCP でも CLI サブコマンドでもよい。CLI ＋ stdout が最も単純で Claude Code ネイティブ。

### 注意：2つの運転モードで保証が違う

| モード | 駆動 | doc10 の不変条件「全 LLM 出力を検証」 |
|---|---|---|
| **PF 駆動**（MVP・反復用） | CLI＋stdout でシステムが手順を指示・エージェントが実行 | **順序はシステムが stdout で握る**（最終実行は agent・人間も見ている） |
| **System 駆動**（後・本番/ヘッドレス） | システムが制御フローを握る | **制御フローで強制**（必ず検証が走る） |

MVP は前者で素早く検証 → 固まったら後者でガード強化、の二段。ツール/ポートは共通なので作り直しは無い。

## 自前実装 vs 委譲の境界（たたき台・要調査）

「どこまで PF に任せ、どこを自前で握るか」の調査用たたき台。
原則は doc10 の境界がそのまま効く：**判断・生成は PF へ委譲、決定的処理と「安全・状態を持つ処理」は自前**でコントロールを握る。

| コンポーネント | 既定 | 自前で得るコントロール | たたき台判断 |
|---|---|---|---|
| L1 レビュー評価（観点違反の発見） | 委譲 | プロンプト/モデル/出力形式・コスト | **委譲**（重い認知処理＝委譲が主目的） |
| L2–L6（矛盾判定/型判定/マージ/草案） | 委譲 | 同上 | **委譲** |
| 構造化出力の保証 | アダプタ | パース安定性・スキーマ強制 | **要調査**（下記グレーゾーン） |
| 自動適用・finding 単位コミット・revert | 自前 | revert 保証・安全・トランザクション | **自前**（PF に黙って編集させない） |
| 決定論的修正の生成（Q21） | 自前ツール | 決定性・再現性 | **自前**（formatter/linter） |
| rule_id 検証/参照除外/仕分け/方向ゲート/provenance | 自前 | 正しさ・監査可能性 | **自前**（[09](09-processing-pipeline.md)⑤ コア） |
| 基準合成・継承マージ・矛盾 hash・警告レジャー | 自前 | 決定性・状態管理 | **自前** |

### グレーゾーン（調査推奨）

- **構造化出力の信頼性** — agentic PF（Claude Code 等）は自由記述寄りで JSON 強制が緩いことがある。
  不安定なら同じポートの裏を **raw モデル API アダプタ（structured output / function calling 対応）** に寄せる。
  PF の階層は〔**agentic PF → raw モデル API → 自己ホスト**〕の3段で、必要な制御に応じて降りる。
- **コスト / レイテンシ / データ持ち出し** — PF 経由は制御しづらい。要件が厳しいなら raw API か self-host。

### 結論（たたき台）

**MVP は agentic PF 委譲＋自前オーケストレーションで始める。** 構造化出力・コスト・データ要件で困ったら
**ポートを保ったまま raw API アダプタへ降りる**。安全/状態系（適用・revert・検証・合成・レジャー）は最初から自前で握る。

## 論点

- **Q22【新規】 PF 能力差の吸収** — 能力宣言の項目と、フォールバックの線引き（最低限どの能力を必須にするか。
  例：**構造化出力は必須**／ファイル適用・ツール実行は任意でシステムが代替）。
- **Q23【新規】 双方向の口（PF→System）の境界と順序ガード** — どの関数を公開するか、
  適用前に検証を強制するか（例：`apply_fix` は検証済みトークンを要求）。**MVP は CLI＋stdout でシステムが順序を指示**するので軽量で足り、本番はツール側ガードで強制。
- **Q5（技術スタック）** — AI レイヤは**外部 PF に委譲**で決着。残るはアプリ本体の実装言語・ホスティング。
