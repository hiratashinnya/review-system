# 05. システム入出力 一覧（概要）

散らばりがちな「システムへの入力 / システムからの出力」を1か所に集約する台帳。
詳細は各リンク先で、ここは**俯瞰と抜け漏れチェック用**。新しい入出力案が出たらまずここに足す。

凡例：✅ 方針あり / 🟡 議論途中 / ⬜ 未着手 ・ 関連未決は `Q#`（[dashboard](../dashboard.md)）。

---

## 入力（システムが受け取るもの）

### A. 評価対象（レビュー1回ごと）

| # | 入力 | 発生源 | 形式（案） | 状態 | 関連 |
|---|---|---|---|---|---|
| I-1 | 評価対象（関連文書群） | 利用者 | コード / 仕様書 / 議事録。**関連ファイルをまとめて1レビュー**。アップロード基本、Git 前提にしない | ✅ | 08 / Q12 |
| I-2 | 文書タイプ**上書き**（任意） | 利用者 | `code` / `spec` / `minutes`。手動指定があれば**推定より優先** | ✅ | 08 / Q4 |
| I-15 | 文書タイプ**推定** | PF / LLM | PF を呼んで得る型候補（＋確信度）。I-2 と **P1.1 で調停**して確定型に（旧 I-2 を発生源で分割・下記「入力の点検」） | ✅ | 08 / Q4 |
| I-3 | 適用スコープ（**明示指定**・任意） | 利用者 / 提出メタ | `org` / `team` / `project` の明示指定。I-11 ルールと **P1.2 で調停**。**MVP は org 単一** | ✅ | 08 / Q13 |
| I-13 | 参照コンテキスト | 利用者 | 上流成果物・前提条件（仕様/要件/アジェンダ等）。**評価対象とは別枠で突き合わせ用**。網羅性/整合性の観点が依存 | 🟡 | 08 / Q19 |

> **発生源が別の入力は調停して使う**：①型＝I-2（利用者の上書き）＋I-15（PF の推定）→ P1.1、②scope＝I-3（利用者の明示指定）＋I-11（org のルール）→ P1.2。
> どちらも「**2 つの別発生源を1つに確定する**」同型のパターン。

### B. 評価基準＝観点（事前に置かれ、継承合成される）

「何を・どういう観点で見るか」という**レビューの中身（ドメイン知識）**。システムの動作設定とは層が違う。

| # | 入力 | 発生源 | 形式 | 状態 | 関連 |
|---|---|---|---|---|---|
| I-4 | 評価基準ファイル群 | 基準メンテナ / org（**系外編集**） | 構造化 Markdown（`criteria/…`）。観点本文＋ルーティング用メタ。`doc_type×scope` で継承・union 合成 | ✅ | 01 / [schema](../schema/README.md) |

### C. システムコンフィグ（システムの振る舞い設定）

観点とは別層の**運用ノブ**。「どう自動化するか・どう通知するか・どうスコープを決めるか」。

| # | 入力 | 発生源 | 形式 | 状態 | 関連 |
|---|---|---|---|---|---|
| I-5 | 自動化ポリシーファイル | 基準メンテナ / org（**系外編集**） | YAML（`policy/…`）。`determinism×severity → 適用モード` | ✅ | 03 / schema |
| I-10 | 通知・エスカレーション設定 | 基準メンテナ / org（config） | 承認者への打ち上げ条件（バッチ/しきい値/dedup 等）。未設計 | ⬜ post-MVP | Q9 |
| I-11 | スコープ決定ルール | 基準メンテナ / org（config） | 文書 → `org/team/project` の対応づけ。**I-3 明示指定と P1.2 で調停**。MVP は org 固定で発火せず | ⬜ post-MVP | Q13 |
| I-12 | 時間/スケジュールトリガ | スケジューラ / システム時計 | 時間・蓄積イベント（観点FB起草・滞留督促）を駆動する時刻入力源。[06](06-event-list.md) で発覚 | ⬜ post-MVP | Q16 |

## 抜け漏れ・要決定（このページ起点）

- I-1 文書の入力粒度（ファイル単位 / PR diff / リポジトリ全体）→ **Q12 起票**。
- I-11 スコープの決まり方（誰がどう指定するか）→ **Q13 起票**。
- 入力を**観点（B）とシステムコンフィグ（C）に分離**済み。観点＝レビューの中身、コンフィグ＝振る舞いノブ。
- O-12 **観点へのフィードバック提案**（システム起草 → 人間が**系外で基準編集** → 次回合成で育成ループ完結）。
- **基準/ポリシー編集は系外＝非イベント（[Q15](06-event-list.md)）**。よって ~~I-8/I-9・E6/E7・O-8/O-10~~ は削除。逸脱・衝突・矛盾は **O-9（合成時警告・Q10）に統合**。
- O-7（Q7 未分類バケツ）はまだ形式未定。
- **[06 イベントリスト](06-event-list.md) で再点検**：時間/異常系イベントの抜けから
  I-12（時間トリガ）・O-13（滞留督促）・O-14（異常系通知）を追加、Q14〜Q17 を起票。
- **[08 intake 設計](08-intake-design.md) で確定**：I-1/I-2/I-3 を MVP 方針で固め、
  欠けていた **I-13 参照コンテキスト（上流成果物・前提条件）** を追加（Q19 起票）。

## 入力の点検（①分割 ②非分割 ③価値接続）

掃除後の入力（I-8/I-9 削除・**I-15 型推定を追加**）を3観点で点検。**発生源は各入力表に追記済み**。

### ① 分けるべきものが分けられているか

| 対象 | 判定 |
|---|---|
| I-1 評価対象 ／ I-13 参照コンテキスト | ✅ 分離済み（**評価する物** vs **突き合わせる物**）。価値経路も別（指摘 vs 網羅性/整合性） |
| I-4 観点 ／ I-5 自動化ポリシー | ✅ 分離済み（**レビューの中身＝ドメイン知識** vs **振る舞いノブ**） |
| I-2 文書タイプ | ✅ **分割実施**：発生源が別（**利用者＝I-2 上書き** ／ **PF＝I-15 推定**）→ P1.1 で**調停**して確定型。アクターが別なら入力も別、が原則 |
| I-6 指摘への判断 | 🟡 **2 経路が同居**：承認→**適用**（O-3）と 却下/修正→**育成**（O-12）。1 入力で価値経路が分岐する点を明示（[process G8](../process/04-gaps-found.md)） |

### ② 区別できないものを分けていないか

| 対象 | 判定 |
|---|---|
| I-4 基準ファイル ／ ~~I-8 上書き提案~~ | 🔴 **同一物**。I-8 は I-4 のファイルを編集するだけで、しかも**系外**＝システムから区別・観測不能 → **I-8 削除**（本掃除で実施） |
| ~~I-9 上書き承認~~ | 🔴 承認＝系外の file マージ。システム入力として存在しない → **削除** |
| I-3 適用スコープ ／ I-11 スコープ決定ルール | ✅ **分離維持**（発生源が別）：**I-3＝利用者の明示指定** ／ **I-11＝org のルール**。二重計上ではなく **P1.2 で調停**して確定 scope（型の I-2/I-15 と同型）。〔前案「I-3 を導出へ格下げ」は撤回〕 |

### ③ 全ての入力は価値に繋がるか（論理の飛躍）

| 入力 | 価値接続 |
|---|---|
| I-1 評価対象 / I-4 観点 / I-5 ポリシー / I-6 判断 / I-7 対象外 / I-13 参照 | ✅ レポート・適用・育成へ直結 |
| I-2 型（上書き） | ✅ 誤判定時に正しい観点を効かせる（手動上書きの価値） |
| I-3 適用スコープ | ⚠ **MVP は org 固定**＝MVP では価値に未接続（team/project は post-MVP）。"入力はあるが効かない"と明示しないと飛躍 |
| I-11 スコープ決定ルール | ⚠ 同上（org 固定下では発火しない・post-MVP） |
| I-10 通知・エスカレーション設定 | ⚠ 出口だった O-8 は削除・O-13 督促は post-MVP → **現スコープで生きた価値経路なし**。MVP では持たない/保留 |
| I-12 時間/スケジュールトリガ | ⚠ 主用途 E13 督促は post-MVP、E10 育成起草はオンデマンドでも可 → MVP では価値が薄い・保留可 |

### 点検の結論・推奨

- **②で削除確定**：I-8 / I-9（系外編集と同一・観測不能）。掃除で削除済み。
- **①で分割実施**：発生源で I-2（型上書き・利用者）と **I-15（型推定・PF）** に分割。I-6 は適用/育成の2経路を明示。
- **②で分離維持（訂正）**：I-3（利用者の明示指定）と I-11（org のルール）は**別発生源**。導出への格下げ案は撤回し、**P1.2 で調停**と整理（型の I-2/I-15＝P1.1 と同型）。
- **③で post-MVP 明示**：I-3 / I-11 / I-10 / I-12 は **MVP では価値未接続**。MVP 入力セットは
  **I-1, I-2(上書き)＋I-15(推定), I-4, I-5, I-6, I-7, I-13** ＋ revert 要求（[I-14 候補](../process/04-gaps-found.md)）。
