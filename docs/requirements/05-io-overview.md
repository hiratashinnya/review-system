# 05. システム入出力 一覧（概要）

散らばりがちな「システムへの入力 / システムからの出力」を1か所に集約する台帳。
詳細は各リンク先で、ここは**俯瞰と抜け漏れチェック用**。新しい入出力案が出たらまずここに足す。

凡例：✅ 方針あり / 🟡 議論途中 / ⬜ 未着手 ・ 関連未決は `Q#`（[dashboard](../dashboard.md)）。

## 恒久ルール（分割の原則）★

> 入出力を分けるか迷ったら、必ずここに立ち返る。
>
> 1. **「もの（実体）＋発生源（外部アクター）」だけで分ける。** **使い道**でも**内部の発生プロセス**でも分けない。
>    - 例：I-6 指摘への判断は、適用にも育成にも使うが**1つの入力**（使い道で割らない）。
> 2. **発生源が PF/LLM のものは入力。** 本システムは **LLM をラップして価値を提供**する。
>    生 LLM 出力＝入力、システムが検証/仕分け/整形した後＝出力（この付加価値が存在意義）。
> 3. **外から見れば内部プロセスが違っても等しく「システム出力」。** もの が同じなら1つ（O-9 統合が好例）、違えば分ける。

## 入力の点検（①分割 ②非分割 ③価値接続）

掃除後の入力（I-8/I-9 削除・**I-15 型推定を追加**）を3観点で点検。**発生源は各入力表に追記済み**。

> **分割の原則**：入出力は「**もの（実体）＋発生源（外部アクター）**」だけで分ける。**使い道**でも**内部の発生プロセス**でも分けない。
> **発生源が PF/LLM のものは入力**（本システムは LLM を**ラップして価値を提供**する）。外から見れば内部プロセスが違っても等しく「システム出力」。

### ① 分けるべきものが分けられているか

| 対象 | 判定 |
|---|---|
| I-1 評価対象 ／ I-13 参照コンテキスト | ✅ 分離済み（**評価する物** vs **突き合わせる物**）。価値経路も別（指摘 vs 網羅性/整合性） |
| I-4 観点 ／ I-5 自動化ポリシー | ✅ 分離済み（**レビューの中身＝ドメイン知識** vs **振る舞いノブ**） |
| I-2 文書タイプ | ✅ **分割実施**：発生源が別（**利用者＝I-2 上書き** ／ **PF＝I-15 推定**）→ P1.1 で**調停**して確定型。アクターが別なら入力も別、が原則 |
| I-6 指摘への判断 | ✅ **分けない**：もの も発生源（レビュアー）も同じで、**使い道が複数（適用／育成）なだけ**。使い道で入力を割ってはいけない |

### ② 区別できないものを分けていないか

| 対象 | 判定 |
|---|---|
| I-4 基準ファイル ／ ~~I-8 上書き提案~~ | 🔴 **同一物**。I-8 は I-4 のファイルを編集するだけで、しかも**系外**＝システムから区別・観測不能 → **I-8 削除**（本掃除で実施） |
| ~~I-9 上書き承認~~ | 🔴 承認＝系外の file マージ。システム入力として存在しない → **削除** |
| I-3 適用スコープ ／ ~~I-11~~ | 🔧 **訂正**：scope の定義は **I-4 観点ファイルのフロントマター**（`scope`/`extends`）。別 config（I-11）として状態化しない → **I-11 削除**。per-review は I-3（任意・MVP=org）の**選択**だけで、型のような2発生源調停は不要 |

### ③ 全ての入力は価値に繋がるか（論理の飛躍）

| 入力 | 価値接続 |
|---|---|
| I-1 評価対象 / I-4 観点 / I-5 ポリシー / I-6 判断 / I-7 対象外 / I-13 参照 | ✅ レポート・適用・育成へ直結 |
| I-2 型（上書き） | ✅ 誤判定時に正しい観点を効かせる（手動上書きの価値） |
| I-3 適用スコープ（選択） | ⚠ **MVP は org 固定**＝MVP では価値に未接続（team/project は post-MVP）。scope の定義自体は I-4 フロントマター |
| ~~I-11~~ | 🗑 削除：別 config/状態にしない（scope は I-4 内）。post-MVP の doc→scope 判定は I-3＋導出で足りる |
| I-10 通知条件設定 | ⚠ O-8/O-13 削除後、残る出口は **O-9 警告の通知チューニング**のみ・post-MVP |
| I-12 時間/スケジュールトリガ | ⚠ 用途は E10 育成起草のみ（**E13 滞留督促は削除**）。育成はオンデマンドでも可 → MVP は薄い・保留 |

### 点検の結論・推奨

- **②で削除確定**：I-8 / I-9（系外編集と同一・観測不能）／ **I-11**（scope は I-4 フロントマターに在り別 config/状態にしない）。
- **①で分割実施**：発生源で I-2（型上書き・利用者）と **I-15（型推定・PF）** に分割。I-6 は分けない（同一物・使い道が複数）。
- **フロントマター項目は別扱いしない**：`scope`/`severity`/`determinism`/`override` 等は I-4 の中身（メタ表は導出）。別ファイルは I-5 ポリシーのみ。
- **③で post-MVP 明示**：I-3 / I-10 / I-12 は **MVP では価値未接続**。MVP 入力セットは
  **I-1, I-2(上書き)＋I-15(推定), I-4, I-5, I-6, I-7, I-13** ＋ revert 要求（[I-14 候補](../process/04-gaps-found.md)）。

## 出力の点検（観点は入力と別）

出力の分割基準は **「もの」**。**内部の発生プロセスで分けない**（外から見れば全て「システム出力」）。
さらに **LLM 生成物は入力、システムが加工して返すものが出力**（ラップ境界）。

### (あ) もの で分ける（内部の発生プロセス違いで分けない）

| 観察 | 判定 |
|---|---|
| O-9 合成時警告（衝突/矛盾/方向） | ✅ 旧 O-8/O-9 を **もの＝「合成時の警告」** として統合済み（発生プロセス違いで割らない好例） |
| O-1 レポート ⊃ O-2/O-3/O-4/O-5/O-7 | 🟡 O-1 は**容れ物（届ける単位）**、O-2〜O-7 はその**中身（別もの）**。**階層**として整理すれば重複でない（O-1＝コンテナと明示済み） |
| O-4 ✋diff ／ O-5 💬原案 | ✅ 別もの（1 修正案 vs 複数案＋要決定）。発生プロセス（P5.3）が同じでも もの が違えば分ける |
| O-3 適用 ／ O-6 revert | ✅ 別もの（前進適用 vs 復元） |

### (い) LLM 生成物は入力、システム加工後が出力（ラップ境界）

| 出力 | raw（発生源 PF＝**入力**） | システムが足す価値（＝**出力**たる所以） |
|---|---|---|
| O-2 指摘 | findings（生） | rule_id 検証・参照除外・仕分け・整形 |
| O-5 原案 | 草案（生） | 提示形へ整形・💬 として束ねる |
| O-7 未分類 | id 無し指摘（生） | 退避・surfacing・新ルール候補化 |
| O-11 ひな形 | 草案（生） | 近傍基準の収集・叩き台整形 |
| O-12 観点FB提案 | 起草（生） | 傾向集計・提案形へ整形 |

→ **生 LLM 出力＝入力、検証/仕分け/整形後＝システム出力**。この付加価値の差こそ**システムが LLM をラップして提供する価値**。raw を素通しするだけの出力は存在意義が無い。

### (う) 全出力が外部アクターに価値として届くか

| 出力 | 届く先 | 価値 | 状態 |
|---|---|---|---|
| O-1〜O-5, O-7 | 利用者 / レビュアー | レビュー結果・修正・判断材料 | ✅ |
| O-6 revert | 利用者 | 取り消しの安心 | ✅ |
| O-9 合成時警告 | 基準メンテナ（と利用者） | 基準の逸脱/衝突の可視化 | ✅ |
| O-11 ひな形 | 基準メンテナ | 立ち上げ高速化 | ✅ |
| O-12 観点FB提案 | 基準メンテナ | 育成 | ✅ |
| ~~O-8 / O-10~~ | — | — | 🗑 削除（系外） |
| ~~O-13~~ / O-14 | — | ~~督促~~ / 異常通知 | 🗑 O-13 削除（滞留は観測不能）／ O-14 post-MVP |

### 出力点検の結論

- **発生プロセスで割らない**：外から見れば全て「システム出力」。もの が同じなら 1 つ（**O-9 統合が好例**）、違えば分ける（O-4/O-5）。
- **ラップ境界を明示**：raw LLM＝入力、加工後＝出力。これがシステムの付加価値。
- **O-1 はコンテナ**（届ける単位）と明記し、中身 O-2〜O-7 と階層で整理。
