# 05. システム入出力 一覧（概要）

散らばりがちな「システムへの入力 / システムからの出力」を1か所に集約する台帳。
詳細は各リンク先で、ここは**俯瞰と抜け漏れチェック用**。新しい入出力案が出たらまずここに足す。

凡例：✅ 方針あり / 🟡 議論途中 / ⬜ 未着手 ・ 関連未決は `Q#`（[dashboard](../dashboard.md)）。

---

## 入力（システムが受け取るもの）

### A. 評価対象（レビュー1回ごと）

| # | 入力 | 形式（案） | 状態 | 関連 |
|---|---|---|---|---|
| I-1 | 評価対象（関連文書群） | コード / 仕様書 / 議事録。**関連ファイルをまとめて1レビュー**。アップロード基本、Git 前提にしない | ✅ | 08 / Q12 |
| I-2 | 文書タイプ | `code` / `spec` / `minutes`。**AI 自動判定＋手動上書き** | ✅ | 08 / Q4 |
| I-3 | 適用スコープ | `org` / `team` / `project`。**MVP は org 単一**（team/project は機構のみ・非露出） | ✅ | 08 / Q13 |
| I-13 | 参照コンテキスト | 上流成果物・前提条件（仕様/要件/アジェンダ等）。**評価対象とは別枠で突き合わせ用**。網羅性/整合性の観点が依存 | 🟡 | 08 / Q19 |

### B. 評価基準＝観点（事前に置かれ、継承合成される）

「何を・どういう観点で見るか」という**レビューの中身（ドメイン知識）**。システムの動作設定とは層が違う。

| # | 入力 | 形式 | 状態 | 関連 |
|---|---|---|---|---|
| I-4 | 評価基準ファイル群 | 構造化 Markdown（`criteria/…`）。観点本文＋ルーティング用メタ。`doc_type×scope` で継承・union 合成 | ✅ | 01 / [schema](../schema/README.md) |

### C. システムコンフィグ（システムの振る舞い設定）

観点とは別層の**運用ノブ**。「どう自動化するか・どう通知するか・どうスコープを決めるか」。

| # | 入力 | 形式 | 状態 | 関連 |
|---|---|---|---|---|
| I-5 | 自動化ポリシーファイル | YAML（`policy/…`）。`determinism×severity → 適用モード` | ✅ | 03 / schema |
| I-10 | 通知・エスカレーション設定 | 承認者への打ち上げ条件（バッチ/しきい値/dedup 等）。未設計 | ⬜ | Q9 |
| I-11 | スコープ決定ルール | 文書 → `org/team/project` の対応づけ（リポジトリ/ユーザー/明示）。未設計 | ⬜ | Q13 |
| I-12 | 時間/スケジュールトリガ | 時間・蓄積イベント（観点FB起草・滞留督促）を駆動する時刻入力源。[06](06-event-list.md) で発覚 | ⬜ | Q16 |

### D. 人間のフィードバック（育成ループ）

| # | 入力 | 形式 | 状態 | 関連 |
|---|---|---|---|---|
| I-6 | 指摘への判断 | 承認 / 修正 / 却下（✋💬 への操作） | 🟡 | 02 / 04 |
| I-7 | 「対象外」フラグ | 🤖 自動修正サマリ流し読み時の違和感マーク | 🟡 | 04 |
| ~~I-8~~ | ~~基準/ポリシー上書き提案~~ | **削除**：I-4 基準ファイルの編集と**同一物・系外**。方向ゲート/矛盾/衝突/locked は**合成時に毎回**判定し警告（O-9） | 🗑 | Q15 |
| ~~I-9~~ | ~~上書き提案への承認/却下~~ | **削除**：承認＝系外の file 編集（システム外） | 🗑 | Q15 |

> 内部処理で LLM を呼ぶ箇所も入出力を持つ（①レビュー時：文書＋合成基準→指摘、②合成前：兄弟ファイル本文→矛盾の有無）。
> 詳細は schema「役割分担」「スコープ内マージ」。ここでは**システム境界の外部 I/O**を主に扱う。

## 抜け漏れ・要決定（このページ起点）

- I-1 文書の入力粒度（ファイル単位 / PR diff / リポジトリ全体）→ **Q12 起票**。
- I-11 スコープの決まり方（誰がどう指定するか）→ **Q13 起票**。
- 入力を**観点（B）とシステムコンフィグ（C）に分離**済み。観点＝レビューの中身、コンフィグ＝振る舞いノブ。
- O-12 **観点へのフィードバック提案**（システム起草 → 人間が**系外で基準編集** → 次回合成で育成ループ完結）。
- **基準/ポリシー編集は系外＝非イベント（[Q15](06-event-list.md)）**。よって ~~I-8/I-9・E6/E7・O-8/O-10~~ は削除。逸脱・衝突・矛盾は **O-9（合成時警告・Q10）に統合**。
- O-7（Q7 未分類バケツ）はまだ形式未定。
- **[06 イベントリスト](06-event-list.md) で再点検**：時間/異常系イベントの抜けから
  I-12（時間トリガ）・O-13（滞留督促）・O-14（異常系通知）を追加、Q14〜Q17 を起票。
- **[08 intake 設計](08-intake-design.md) で確定**：I-1/I-2/I-3 を MVP 方針で固め、
  欠けていた **I-13 参照コンテキスト（上流成果物・前提条件）** を追加（Q19 起票）。

## 入力の点検（①分割 ②非分割 ③価値接続）

掃除後の入力 `I-1〜I-7, I-10〜I-13`（I-8/I-9 は削除）を3観点で点検。

### ① 分けるべきものが分けられているか

| 対象 | 判定 |
|---|---|
| I-1 評価対象 ／ I-13 参照コンテキスト | ✅ 分離済み（**評価する物** vs **突き合わせる物**）。価値経路も別（指摘 vs 網羅性/整合性） |
| I-4 観点 ／ I-5 自動化ポリシー | ✅ 分離済み（**レビューの中身＝ドメイン知識** vs **振る舞いノブ**） |
| I-2 文書タイプ | 🟡 **要分離**：入力の実体は「**型の手動上書き（任意）**」だけ。型そのものは AI 推定からの**導出値**。"入力"と"導出"が同居 → 「型上書き（入力）」と「型（導出）」を区別すべき |
| I-6 指摘への判断 | 🟡 **2 経路が同居**：承認→**適用**（O-3）と 却下/修正→**育成**（O-12）。1 入力で価値経路が分岐する点を明示（[process G8](../process/04-gaps-found.md)） |

### ② 区別できないものを分けていないか

| 対象 | 判定 |
|---|---|
| I-4 基準ファイル ／ ~~I-8 上書き提案~~ | 🔴 **同一物**。I-8 は I-4 のファイルを編集するだけで、しかも**系外**＝システムから区別・観測不能 → **I-8 削除**（本掃除で実施） |
| ~~I-9 上書き承認~~ | 🔴 承認＝系外の file マージ。システム入力として存在しない → **削除** |
| I-3 適用スコープ ／ I-11 スコープ決定ルール | 🟡 **二重計上の疑い**：scope は I-11（ルール）＋文書から**導出**される値。I-3 を独立入力として併記すると「導出値」と「ルール」が重複。入力は **I-11（＋任意の明示指定）に寄せ、I-3 は導出**と整理すべき |

### ③ 全ての入力は価値に繋がるか（論理の飛躍）

| 入力 | 価値接続 |
|---|---|
| I-1 評価対象 / I-4 観点 / I-5 ポリシー / I-6 判断 / I-7 対象外 / I-13 参照 | ✅ レポート・適用・育成へ直結 |
| I-2 型（上書き） | ✅ 誤判定時に正しい観点を効かせる（手動上書きの価値） |
| I-3 適用スコープ | ⚠ **MVP は org 固定**＝MVP では価値に未接続（team/project は post-MVP）。"入力はあるが効かない"と明示しないと飛躍 |
| I-11 スコープ決定ルール | ⚠ 同上（org 固定下では発火しない・post-MVP） |
| I-10 通知・エスカレーション設定 | ⚠ 出口だった O-8 は削除・O-13 督促は post-MVP → **現スコープで生きた価値経路なし**。MVP では持たない/保留 |
| I-12 時間/スケジュールトリガ | ⚠ 主用途 E13 督促は post-MVP、E10 育成起草はオンデマンドでも可 → MVP では価値が薄い・保留可 |

### 点検の結論・推奨

- **②で削除確定**：I-8 / I-9（系外編集と同一・観測不能）。本掃除で削除済み。
- **①で要分離**：I-2 を「型上書き（入力）／型（導出）」に、I-6 の 2 価値経路（適用/育成）を明示。
- **②③で要整理**：I-3 は I-11 から**導出**（独立入力でない）。scope の入力は I-11（＋明示指定）に一本化を推奨。
- **③で post-MVP 明示**：I-3 / I-11 / I-10 / I-12 は **MVP では価値未接続**。MVP の入力セットは
  **I-1, I-2(上書き), I-4, I-5, I-6, I-7, I-13** ＋ revert 要求（[I-14 候補](../process/04-gaps-found.md)）に絞れる。
