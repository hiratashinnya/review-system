# 10. LLM ⇄ システム境界（イベントリスト）

LLM が呼ばれる箇所を**全部洗い出し**、各点で「LLM がやること（判断・生成＝確率的）」と
「システムがやること（機構・検証・強制＝決定的）」を分ける。2軸（機械判定／運用）をアーキテクチャに適用したもの。

## 境界の原則

- **LLM＝判断と生成**：観点違反の発見・矛盾の有無・型推定・マージ案・原案/草案。**確率的＝あくまで助言**。
- **システム＝決定的機構**：スキーマ/`rule_id` 検証・仕分け（ルーティング）・参照除外・適用/コミット/revert・`locked` 機械拒否・ゲート強制。
- **不変条件：すべての LLM 出力は、システムの検証 か 人間のフォールバックを必ず通る。** LLM 単独で確定する経路は作らない。
  → これが「LLM を過信しない」の構造的担保。LLM が間違えても後段で機械的に拾うか、人へ逃がす。
- **LLM に渡さないもの**：routing 用メタ（`severity`/`determinism`/`override`）。仕分けはシステムの仕事（[07](07-ai-input-design.md) 原則）。

## LLM ⇄ システム イベント一覧

| # | トリガ（システム→LLM） | LLM の仕事（確率的） | 返り値 | システムの仕事（決定的） | 迷い/失敗のフォールバック |
|---|---|---|---|---|---|
| L1 | 文書提出→プロンプト構築 | 観点違反の発見・`rule_id` 付与・修正原案生成 | findings / unmatched（`location.file` 必須） | `rule_id` 検証・参照除外・仕分け・自動適用（[09](09-processing-pipeline.md)） | 検証外れ/自己申告→❓未分類→人（Q7） |
| L2 | 兄弟ファイルの同 id 本文（合成前） | **本文が矛盾しているかだけ**判定 | 矛盾 yes/no（＋理由） | 矛盾なし→共存／矛盾あり→親フォールバック（Q10） | 矛盾→人へ警告 |
| L3 | 文書提出 | 文書タイプの推定＋確信度 | type 候補 | 既定採用し基準をルーティング（[08](08-intake-design.md)） | 低確信/誤り→ユーザー手動上書き（Q4） |
| L4 | 同一箇所への重複 fix を検出（[09](09-processing-pipeline.md)⑦） | **衝突のマージ案生成（2段構えの1段目）** | 統合 fix or「迷い」フラグ | finding 単位コミットの適用・順序管理（Q3） | 迷い→**人が決定（2段目）**（Q20） |
| L5 | 新 doc_type/スコープ立ち上げ | 基準（観点）の草案生成 | criteria 叩き台 | 保存し以後の合成に投入（O-11） | 人がレビュー |
| L6 | フィードバック蓄積（時間/しきい値） | 基準/ポリシー変更の原案起草 | 変更提案 | 方向検出・surfacing（O-12） | 人が承認（I-9） |

## 読み取れること（境界の健全性チェック）

- **L1〜L6 すべてに「システム検証」か「人フォールバック」が付いている** → 不変条件を満たす。LLM 単独確定の経路は無い。
- パターンは一貫：**LLM は前に出て案を出し、システムが機械で受け止め、迷いは人へ逃がす。**

## 通して見えた論点

- **Q20 確定（2段構え）** — 自動修正衝突は ①LLM がマージを試み、②迷うところだけ人が決める。
  「迷い」を LLM に申告させ（L4 の返り値）、システムがそれを人へルーティングする。
- **Q21【新規】 決定論的な修正の生成主体** — lint/format/誤字のように**答えが一意な修正**は、
  LLM 生成だと 🤖 自動適用でも確率的揺れが残る。こうした修正は **LLM でなく決定的ツール（formatter/linter 等）が生成**すべきでは？
  境界の最適化（「決定論的＝システム生成 / 判断含む＝LLM 生成」）。
