# ドキュメントシステム設計

> **目的**：「実装が変わったのに設計書に反映し忘れる」ドキュメント腐敗を、人の記憶に頼らず撲滅する。
> 各文書要素に ID を付与し、上流への参照辺（エッジ）に関係種別＋ステータスを持たせ、依存グラフを機械生成・点検する。
>
> **背景**：[Issue #5](https://github.com/hiratashinnya/review-system/issues/5)
>
> **スコープ**：このディレクトリはシステムの「設計」。実装完了後に review-system 本体のドキュメントへ適用する。

## 設計の進め方

```
01 文書項目セット確定
  ↓
02 メタ属性スキーマ設計（id 体系・エッジ属性・版管理）
  ↓
03 接続要否マトリクス（全段記載必須）
  ↓
04 Markdown 埋め込み記法決定（<details> トグル）
  ↓
05 検証ツール実装（段階①②③ → sub-issue で④）
  ↓
review-system 本体ドキュメントへ適用
```

## 決定サマリ（本議論で確定した事項）

| # | 論点 | 決定 |
|---|---|---|
| Q-新1 | 価値/SR/仕様の境界 | 3層に切る・**ファイルも分離**（VAL/SR/FR 別ファイル） |
| Q-新2 | NFR の独立 | **独立させる**（`NFR-` プレフィックス） |
| Q-新3 | 用語/データ辞書の独立 | **独立させる**（`TERM-`）。リンク必須対象の絞り込みは 05 で後日 |
| Q-新4 | ドキュメント向け検証の範囲 | **機械検証可能なものはすべて実施**。段階化して sub-issue 可 |
| Q-新5 | 意思決定/QA/ペンディングの統合 | **id 通貫・ライフサイクルはノードのステータスで区別**。エッジには関係種別＋**伝播ステータス**（反映済/未反映/対象外）を付与 |
| C2 | コード⇔設計の構造比較 | 段階①②③先行・④は sub-issue 化 |
| C3 | トレース対象集合の確定タイミング | 検証器実装着手前に必須（06 で詰める） |
| D1 | id の形式 | **連番**（`PREFIX-NNN`）・永続・意味は title が持つ |
| D2 | 多段省略の可否 | **全段記載必須**（推移上流すべてに辺） |
| D3 | SRC/TC の全段記載 | 辺爆発のため**直接先のみ**を推奨（[03 §4](03-connection-matrix.md) 要確認） |
