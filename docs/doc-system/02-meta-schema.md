# メタ属性スキーマ

> ノード（意味要素）とエッジ（上流参照辺）が持つ属性の定義。
> 可視性トグルの仕組みと埋め込み記法は [03](03-notation.md) で確定。
> ここでは**何を持つか**だけを決める。

---

## 1. 可視性トグルの方針

<details><summary>⬡ 暫定決定 DD-S01 · v1.0</summary>

```yaml
id: DD-S01
type: DD
version: "1.0"
status: decided
edges:
  - { to: DD-S01, kind: affects, status: done }
```
</details>

### 論点
メタ属性をインラインに置きつつ、通常閲覧時に邪魔にならない仕組みが必要。

### 選択肢

| 案 | 仕組み | レンダリング | 機械可読 | 追加ツール |
|---|---|---|---|---|
| **A: `<details>/<summary>`** | HTML の折りたたみ要素 | GitHub ネイティブでトグル可（**既定で折り畳み**） | `<details>` ブロックを正規表現で抽出 | 不要 |
| B: HTML コメント | `<!-- @node ... -->` | 常に不可視 | コメント抽出 | 不要 |
| C: 不可視版を自動生成 | ソースはインライン注釈・ツールが stripped 版を生成 | 生成ファイルは散文のみ | ソースを解析 | 生成ツール必須 |

### 推奨・暫定決定
**A（`<details>/<summary>`）を採用**し、**C（stripped 版自動生成）を後続オプションとして保持**。

根拠：
- GitHub Markdown でネイティブにトグル可。追加ツール不要で今すぐ機能する。
- `<summary>` 行に `⬡ ノードID · vX.Y` を置くと、折り畳み状態でも ID が見える。
- stripped 版（prose のみ）が欲しくなった時点で `<details>...</details>` ブロックを除去するスクリプト（1関数）を追加すればよい——先に実装する必要はない。

影響範囲：
- `<details>` ブロックのパース仕様を 03 で確定する必要がある。
- ソースが `<details>` を含む Markdown になるため、grep 等で `<details>` ブロック内の YAML を無視する配慮が必要。

## 2. ノード属性スキーマ

各意味要素（1ノード）が持つ属性。

```yaml
# 必須
id:      string   # グローバル一意。"PREFIX-識別子"（例: DM-ReviewReport, FR-09）
type:    string   # 要素型（下記 §4 の型リスト）

# 推奨
version: string   # "MAJOR.MINOR"。構造/型変化=MAJOR・内容のみ=MINOR

# 条件付き
status:  string   # 意思決定ノード(DD/Q/PEND)のみ: open|decided|deferred|closed
mvp:     string   # PR8 印が要る要素のみ: in|out
```

### 版（version）の意味

| 変化 | MAJOR上げ | MINOR上げ |
|---|---|---|
| フィールド追加・削除・型変更 | ✅ | — |
| ポート/シグネチャ変更 | ✅ | — |
| 文言・説明のみの変更 | — | ✅ |
| 観点の追加（本文のみ） | — | ✅ |

> **MAJOR が上がった**ノードへの `refines`・`realizes`・`verifies` 辺の `status` は
> 自動的に `pending` へリセットする（検証ツールが検出・報告）。

## 3. エッジ属性スキーマ

ノードが持つ上流参照辺（ノード → 依存先）のリスト。

```yaml
edges:
  - to:     string   # 参照先ノード id（必須）
    kind:   string   # 関係種別（必須・下記 §5）
    status: string   # 伝播ステータス（必須・下記 §6）
    note:   string   # 備考・理由（任意）
```

> `edges` は**このノードが依存する上流**を列挙する（下流から上流への辺）。
> 下流一覧はグラフ走査で求め、ドキュメントには書かない（二重管理を避ける）。

## 4. 要素型（type）一覧

| 型値 | 層 | 説明 |
|---|---|---|
| `VAL` | Why | 価値命題 |
| `SR` | Why | ステークホルダー要求 |
| `FR` | What | 機能仕様 |
| `NFR` | What | 非機能・制約 |
| `TERM` | 共有語彙 | 用語・データ辞書エントリ |
| `ACTOR` | 分析 | 外部アクタ |
| `I` | 分析 | 入力（もの＋発生源） |
| `O` | 分析 | 出力（もの＋受け手） |
| `P` | 分析 | 論理プロセス |
| `E` | 分析 | イベント |
| `ORC` | 設計・振る舞い | オーケストレーション段 |
| `DS` | 設計・振る舞い | 状態・データストア |
| `MOD` | 設計・静的 | モジュール |
| `DM` | 設計・静的 | ドメイン型 |
| `PORT` | 設計・静的 | ポート・アダプタ |
| `PRS` | 設計・静的 | 永続設計 |
| `SCM` | 設計・静的 | スキーマ |
| `TC` | 検証 | テストケース |
| `DD` | 横断 | 意思決定（決定済み） |
| `Q` | 横断 | 未決論点 |
| `PEND` | 横断 | 先送り論点 |

## 5. エッジ関係種別（kind）

| 値 | 意味 | 典型的な from → to |
|---|---|---|
| `refines` | 上位概念を詳細化する（層間の主辺） | FR → SR、DM → TERM、P → FR |
| `realizes` | 設計・仕様をコードが実現する | ソース → DM/PORT/ORC |
| `verifies` | テストが要求・コードを検証する | TC → FR、TC → ソース |
| `affects` | 意思決定が任意の要素に影響する | DD/Q → 任意 |

> **`affects` 辺のドリフト検出**：`DD` / `Q` ノードで `status: decided` かつ
> 任意の `affects` 辺の `status: pending` → **反映漏れ確定**。

---

## 6. エッジ伝播ステータス（status）

| 値 | 意味 |
|---|---|
| `pending` | 上流の変化がこの辺の先にまだ反映されていない |
| `done` | 反映済み |
| `n/a` | トレース上辺は存在するが、伝播反映は不要（影響なしと確認済み） |

### ステータス遷移

```
[新規辺作成]  →  pending
    │
    ├─ 反映完了  →  done
    └─ 影響なしと確認  →  n/a

[上流ノードの MAJOR 版上げ]
    → refines / realizes / verifies 辺を done → pending へリセット
      （検証ツールが自動検出・報告。手動でリセットしてもよい）
```

### 意思決定ノードのライフサイクル（status on node）

```
Q(open) ─決定─→ DD(decided) ─クローズ─→ DD(closed)
Q(open) ─保留─→ PEND(deferred) ─再開─→ Q(open)
```

> id は通貫（ライフサイクルが変わっても id は変えない）。
> `affects` 辺は `DD(decided)` になった時点で各辺の `status` を `pending` に設定し、
> 伝播が完了した辺から順に `done` または `n/a` へ更新する。

## 7. ファイル構造との対応

ファイルはコンテナ（ノードではない）。ファイル自体のメタは frontmatter に置く。

```yaml
---
# ファイルメタ（任意）
doc_title: クラス設計
doc_version: "3.0"    # ファイル全体の版（変更履歴の目安・ノード版とは独立）
---
```

ファイル内の各意味要素が1ノード。`<details>` ブロックに YAML で記述（03 で記法確定）。
