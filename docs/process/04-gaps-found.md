# プロセス設計 04 — 発見した漏れ・論点

プロセス分解で炙り出した、05/06 等の漏れ・責務配置の論点。
**矛盾（既存決定と両立しない事実）は無し**＝停止案件なし。以下はすべて漏れ・配置・視点の整理。

## 解消済み（この設計内で確定）

| # | 内容 | 解消 |
|---|---|---|
| G1 | L1 図に **P1→PF（文書タイプ判定）** の線が欠落していた | [01](01-dfd-level1.md) 図に P1↔PF を追加。型判定も PF 呼び出し（[10](../requirements/10-llm-system-boundary.md) L3） |
| G2 | **方向ゲート判定の置き場**が曖昧だった | **P2 合成**に確定（[schema](../schema/README.md) のマージ時 reject と一致）。育成 P6 に置くと二重化するので置かない |
| G3 | **警告発行（既出判定）**が P2 合成と P6 編集の両方から呼ばれる cross-cutting | **P6.5 警告発行**として単一責務の共有プロセスに切り出し（DS4 警告レジャーを P2/P6 双方が使う） |
| G6 | **同一箇所の複数 fix 衝突（Q20）**の処理が P5 に明示されていなかった | **P5.2** に衝突解決（Q20 2段構え）を組み込み |

## 台帳へ反映推奨（漏れ）

| # | 内容 | 推奨アクション |
|---|---|---|
| G4 | **revert 要求**が 05 の入力（I-#）に無い（出力 O-6 だけ）。プロセス上は利用者→P5.4 の明確な入力 | 05 に `I-14 revert 要求` を追加 |
| G5 | **PF を 05 では内部処理**として外部 I/O に数えていなかったが、本設計では**外部エンティティ** | 矛盾ではない（05 が明示的に LLM I/O を後回しと注記）。05 に「PF＝外部エンティティ」の一言を足すと整合が明確 |

## 横断（別 Q で継続）

| # | 内容 | 扱い |
|---|---|---|
| G7 | **異常系**（合成失敗・PF 障害・パース失敗・空文書）は DFD のデータフローに現れない横断的エラー経路 | [Q17](../dashboard.md) で degrade／エラー通知(O-14) として設計。各 L2 プロセスに「失敗→O-14」が付く |

## まとめ

- **停止すべき矛盾は出なかった。** 出たのは漏れ（G1/G4/G5）・責務配置（G2/G3）・既知 Q への接続（G6/G7）。
- プロセス分解は 09 パイプラインと整合（P1=intake, P2=合成, P3=評価, P4=検証仕分け, P5=適用レポート, P6=育成）。矛盾なく**より細かい責務**に割れた。
- 台帳反映推奨は G4（I-14 追加）と G5（PF 外部エンティティ明記）。指示があれば 05 に反映する。
