# プロセス設計 05 — イベント総点検（発生源｜情報｜受け手）

> 目次：[00 コンテキスト](00-context.md)／[01 L1](01-dfd-level1.md)／[02 分解](02-decomposition.md)／[03 状態](03-state-inventory.md)／[04 漏れ](04-gaps-found.md)／**05 総点検**

[06 イベントリスト](../requirements/06-event-list.md)の各イベントを、**入力から内部プロセスを通って出力へ**繋がるか、
`|情報の発生源|情報|情報の受け手|` の三つ組で1ホップずつ追う。**価値経路が途切れていないか**の最終点検。

## 点検のルール（階層をまたがない）

- **外部エンティティ／データストアは L1 プロセスの境界にしか繋がない。** リーフへは L1 を経由する。
  - ✅ `利用者 → P1 → P1.1`　／　🚫 `利用者 → P1.1`（直結禁止）
- **リーフから出る時も L1 を経由**してから次へ。
  - ✅ `P1.1 → P1 → P2 → P2.1`　／　🚫 `P1.1 → P2.1`
- 凡例：外部＝利用者/レビュアー/基準メンテナ/**PF（LLM＝外部）**。ストア＝DS1 基準 / DS2 矛盾キャッシュ / DS3 finding-commit / DS4 警告レジャー / DS5 フィードバック。

## E14 異常系 → O-14（Q17 確定・横断 supervisor）

**入力**：各段の失敗（基準パース失敗・スコープ未解決・LLM 障害・空文書）　**出力**：O-14 異常系エラー通知（空文書のみ no-op）

異常系は正常データフローではなく**全段にまたがる横断経路**なので、`発生源（失敗した L1 プロセス）｜情報（失敗事由）｜受け手` で1ホップに畳む（[00 横断モデル](00-context.md#異常系の横断モデルq17-確定)・要件 [13 S3](../requirements/13-stabilization.md)）。

| 情報の発生源 | 情報 | 情報の受け手 |
|---|---|---|
| P2（基準パース失敗 / スコープ未解決） | fail-close 事由（ファイル/理由） | P5（レポート経路）|
| P3（LLM 障害＝retry 後も不可） | fail-close 事由（障害段）＋**⑦自動適用を抑止** | P5 |
| P3.3/P4.1（LLM 出力不正） | degrade＝❓未分類化（[S1](../requirements/13-stabilization.md)・既設） | P5（❓区分）|
| P1.3（空文書＝対象ゼロ） | benign no-op（0件レポート・O-14 出さない） | 利用者 |
| P5 | **O-14 異常系エラー通知**（実行可能な次手つき） | 利用者 / 基準メンテナ |

> ⑦自動適用（P5.2）は ④/⑤ クリーン完了が前提。上流 fail-close 時は **DS3 へ一切コミットしない**（[S4](../requirements/13-stabilization.md)）。

✅ E14：失敗→fail-close→O-14 まで連続（誤った部分結果を出さず可視化）。空文書のみ遮断せず no-op。

## 総点検の結論

- **全イベント（E1〜E5, E8〜E10, E14）で入力→プロセス→出力が連続**。階層スキップ無し（外部/ストアは必ず L1 経由でリーフへ）。
- **削除済**：E6/E7（系外編集）・**E13/O-13（滞留はシステムが観測不能）**。**Q17 確定**：E14 異常系（O-14）は fail-close＋横断 supervisor としてモデル化（明示は MVP・degrade は F16）。
- ラップ境界どおり、PF 生成物（型推定/findings/草案）は**入力**として入り、検証/整形後に**出力**へ出ている。
