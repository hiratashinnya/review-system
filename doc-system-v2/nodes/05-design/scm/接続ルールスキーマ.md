config.yaml の必須接続セクション（`must_link_to`＝依存辺／`must_be_linked_from`＝被依存辺）の構造スキーマ。RULE-006（必須辺欠如）が config 駆動で判定する各規則行の形を定義する。同一 node の複数行は AND、target/source の配列は OR を意味する。

**フォーマット**: YAML（マップのリスト×2 セクション＋シンボル適格性マップ×1）
**必須フィールド**:
- `must_link_to`：規則行リスト。各行 `{ node, target, activate_stage, severity, reason }`
  - `node`（型名・規則の主体）・`target`（依存先の型名。配列なら OR）・`activate_stage`（発火開始ステージ・stages のいずれか）・`severity`（`error` | `warning`）・`reason`（説明文字列）
- `must_be_linked_from`：規則行リスト。各行 `{ node, source, activate_stage, severity, reason }`
  - `node`（型名）・`source`（被依存元の型名群。配列＝OR）・`activate_stage`・`severity`・`reason`
- `src_symbol_eligibility`（DD-10）：設計種別（型名）→ 許容する `src.kind`（シンボル種別）配列のマップ。SRC⇄設計ノードのリンク充足を「設計種別ごとに張ってよいシンボル種別」で機械判定するための表。施行器(#163)が `must_link_to`（src→X）と `must_be_linked_from`（X←src）のカウント時に SRC の `source.kind` を本マップで照合し、適格 kind のみ有効カウントする（例＝MOD 担体はモジュール単位のシンボルにのみ対応可・関数/クラス流用での誤充足を排除）。
  - キー＝設計種別の型名（`mod` / `dm` / `port` / `orc` / `prs` / `prompt` / `cfg`）・値＝許容 `src.kind` の配列（OR）。
  - 現行マップ（config.yml と一致）：
    - `mod: [module]`（モジュール担体＝モジュールレベルのみ）
    - `dm: [class]`（型定義＝クラス／dataclass・enum）
    - `port: [class]`（ポート＝クラス／Protocol・ABC）
    - `orc: [function]`（制御フロー入口＝関数）
    - `prs: [class, function]`（永続化実装＝クラス or 関数）
    - `prompt: [file]`（プロンプト資産＝skill/agent ファイル・正規パス規約一致）
    - `cfg: [file]`（設定インスタンス＝config ファイル・正規パス規約一致）
  - 判定粒度（DD-10）：論点1（sprint-1）は `source.kind` 一致で確定（DM/PORT の class 重複は許容）。class の厳密化（Protocol/dataclass 判定）と多言語化は PEND-a。`prompt`/`cfg` の `file` 適格性は「存在＋正規パス規約一致」まで機械判定し、中身の意味判定完全機械化は PEND-b。

**注記（source.kind 語彙拡張・#163）**: 本マップが参照する SRC の `source.kind` 語彙は、施行器(#163・Phase B)実装で `{module, class, function, method, file}` へ拡張される（現行より `module`・`file` を追加）。現時点の config 反映は inert（施行は #163 で行う）。
