**論点（Issue #78 論点2）**: condition の5値セット `normal / boundary / empty / failure / error`（等価分割）は今も適切か。特に `empty`（空・ゼロ件・null）を `boundary` から独立させた経緯（**FND-13**・resolved）を踏まえ、独立を維持するか `boundary` へ統合（＝4値化）するか。

**現状**:
- `config.yml` の condition_vocab は `[normal, boundary, empty, failure, error]`。各語のセマンティクスはコメントで定義: normal=正常系 / boundary=境界値 / empty=空・ゼロ件・null / failure=仕様違反を正しく検出（sad-path）/ error=処理不能な異常入力（fail-close 対象）。
- `empty` は **FND-13**（v0.1.1・resolved）で追加された。指摘は「空集合・ゼロ件・null・未設定を表す独立クラスが無く、normal/boundary に混ぜると等価分割が崩れテスト設計の網羅穴になる」。in-graph アンカー SPEC-31（in-graph 0 件で RULE 評価を全スキップ）が `empty` を実使用しており、SPEC-31 → FND-13 のバックリファレンス辺で紐づく（辺逆転完了済み）。
- したがって `empty` は既に実利用があり、単純削除はできない（少なくとも SPEC-31 の condition 再割当が必要）。

**選択肢**:
- **A（5値維持）** — 現行セットを妥当と確定し変更しない。`empty` は FND-13 の根拠（空集合はゼロ件境界とは別の等価クラス。null/未設定/0件は境界値の「端」ではなく「集合そのものが空」で、テスト設計上独立に扱うべき）で正当化される。トレードオフ＝5値の維持コスト（各 author・notation・テスト戦略が5値を列挙し続ける）。
- **B（empty を boundary へ統合し4値化）** — `empty` を廃し `boundary` に吸収、SPEC-31 等の empty 使用ノードを boundary へ再割当。トレードオフ＝FND-13 で解消した「空集合の網羅穴」が再発する懸念。境界値（値域の端）とゼロ件（集合が空）を同一クラスにすると等価分割の粒度が粗くなり、テスト設計で空集合ケースが暗黙化しやすい。FND-13 を意図的に覆す決定になるため、覆す根拠が要る（CLAUDE.md「意図的設計の尊重」）。
- **C（別軸で再編・empty を normal 側の下位に置く等）** — 空集合が「正常なゼロ件」か「異常な欠落」かで意味が割れる点に着目し、empty を用途で分割/再配置する。トレードオフ＝語彙が増える方向で複雑化。論点1（傘表現）・論点3（always_error）と絡み、5値の等価分割の単純さを損なう。

**推奨**: **A（5値維持）**。理由：(1) `empty` は FND-13 で「空集合＝境界とは別の等価クラス」という明確な根拠のもと追加され、SPEC-31 で実利用がある意図的設計である。覆すには FND-13 の根拠を否定する新たな根拠が必要だが、現時点でそれは無い（CLAUDE.md「起票前に違反が意図的・documented な設計でないか確認・覆さない」）。(2) B は解消済みの網羅穴を再発させるリスクがあり、削減効果（5→4）に見合わない。(3) C は等価分割の単純さを損なう。**5値の妥当性を追認する決定でよいが、これはオーナー判断**（Issue #78 が明示的に問うている論点のため追認も決定として残す）。

**論点4（整合連動）への波及**: vocab セットを変える場合（B/C）、coverage_rules（`fr.required_conditions=[normal]` / `recommended_conditions=[failure,error]`）・RULE-016（語彙外検査）・各 author の語彙列挙を同期改訂する必要がある。詳細は論点4 の Q で扱う。A（維持）ならこの波及は生じない。

**他論点との結合**: 本 Q（論点2）は vocab セットの定義そのもの。論点1（傘）・論点3（always_error）は「個々のノードにどの値を割り当てるか」の運用で、セット定義が前提になる。したがって論点2 が B/C なら論点3 の選択肢（error/failure 統一）も影響を受ける。本 Q は独立の決定軸（セットの構成）のため別 Q とした。

**ブロッカー**: 5値セットの妥当性は**オーナー判断**。実施時期も含めてオーナーが決定する（**独断でのスプリント繰越は禁止**・`scheduled` 空で判断を仰ぐ）。決定後は DD へ昇格。

**指摘時 ref_version**: condition 属性なし・語彙外（RULE-016）"0.1"（同 SPEC サイドカー v0.1.0 時点）

**決定（2026-07-06 オーナー承認）**: 選択肢A（5値維持）を採用。本 Q は DD「condition の5値語彙セットを維持（語彙妥当性 Q より昇格）」として昇格・決定済み。`empty` を `boundary` へ統合せず、`normal / boundary / empty / failure / error` の5値語彙セットをそのまま維持する。
