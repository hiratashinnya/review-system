**改訂（MINOR バンプ v0.1→v0.2・G3 スコープ追加）**: Q-2（リーフプロセスの SPEC 1:1 不足＝傘 SPEC 暫定マップの要件層細分化判断）の射程に、**グラフ網羅性点検アンブレラ SPEC-29 配下のリーフ P マップ**を追加する。義務辺 `→SPEC-29`（ref_version "0.1"・SPEC-29 現バッジ v0.1）を追加し、SPEC-29 配下のリーフマップ判断が Q-2 の射程であることを義務辺で明示する。Q-2 が open（未決）である間は RULE-002 WARNING が `→SPEC-29` にも及ぶが、これは「SPEC-29 配下のリーフマップが未決でオーナー判断待ち」であることの正しい表明であり、決定（DD 昇格）時に義務辺を解消する。

**G3 の状況**: 現状、親 P-3-1（グラフ網羅性点検）が傘 **SPEC-29**（normal・アンブレラ）を参照し、その子リーフ P-3-1-1（未駆動出力検出）・P-3-1-2（未定義反応イベント検出）・P-3-1-3（未消費入力検出）は **SPEC-30-x**（分析ノード接続漏れ検出 failure 系）のみを担当している。一方、SPEC-29 の子アサーション **SPEC-29-1（孤立・必須辺欠如のゼロ件通過・normal）**・**SPEC-29-2（価値経路到達の充足判定・normal）** は、どのリーフ P からも個別に 1:1 参照されておらず、**親 P-3-1 が傘 SPEC-29 を参照することで transitively（傘マップ経由で）含まれる暫定状態**にある。これは Q-2 が扱う「リーフ P が既存 SPEC に 1:1 対応を持たず傘 SPEC へ暫定マップされている」構図と同一であり、推奨 A（傘マップ維持・実害が出た時点で細分化）と整合する。

**論点**: 分析層の DFD レベリング深化で生じたリーフプロセス（P-5-1/5-2/5-3・P-1-6・P-2-5・P-4-1/4-3 等）が、既存 SPEC に 1:1 対応を持たず、暫定で**傘 SPEC**へマップされている（設計 §5.3 一覧）。具体的な傘マップ:
- **P-5-1 config 読込／P-5-2 スキーマ検証／P-5-3 設定スライス組立** → 傘 **SPEC-21**（読込単体・config 自身の壊れ・スライス射影の SPEC が無い）。
- **P-1-6 検査ビュー射影** → 傘 **SPEC-1**（射影スライス生成の SPEC が無い・新概念）。
- **P-2-5 抑制・発火フィルタ** → SPEC-20/21 系（抑制を独立プロセス化する SPEC は無い・挙動 SPEC はある）。
- **P-4-1 違反統合／P-4-3 G 番号付与** → 傘 **SPEC-25**（統合単体・G# 採番の独立 SPEC が無い）。

これらのリーフは「既存の最も具体的な SPEC」へ暫定マップしたが、傘 SPEC のままにするか、要件層で SPEC を細分化（リーフ単位の SPEC を新設）するかは要件層への波及であり、オーナー判断を要する。

**選択肢**:
- **A（当面は傘マップ維持・実害が出た時点で細分化）** — 既存の傘 SPEC へのマップを当面維持し、各リーフ P→SPEC 辺（必須・RULE-006）は傘 SPEC で満たす。テスト設計・実装で「傘 SPEC では一意にテストできない／責務が曖昧」という実害が顕在化した時点で、対象リーフのみ SPEC を細分化（`-N` 採番の子 SPEC 新設）する。要件層の即時改訂を避け、分析層の改訂を先に確定できる。トレードオフ＝傘 SPEC が複数リーフを束ねる間はカバレッジ計測の粒度が粗いまま残る。
- **B（要件層で SPEC を即時細分化）** — 暫定マップした全リーフに対応する SPEC を要件層（spec.md）で新設し、リーフ P→子 SPEC の 1:1 対応を即時に確立する。トレードオフ＝要件層の大量改訂（新規 SPEC 著作＋FR への接続＋カバレッジ規則更新）が発生し、分析層改訂と並行して要件層フェーズの作業が膨らむ。スタンプ結合解消（分析層の主目的）と直接関係しない要件層改訂を巻き込む。
- **C（傘マップを正式仕様として固定・細分化しない）** — 傘 SPEC への複数リーフ集約を正式な設計として固定し、SPEC は細分化しない方針を確定する。トレードオフ＝リーフごとのテスタビリティ・カバレッジ追跡が永続的に傘単位に留まり、FND-2/FND-19 等で進めてきた「単一責務＝単一 SPEC」方向と逆行する。

**推奨**: **A（当面は傘マップ維持・実害が出た時点で細分化）**。理由：(1) 本パスの主目的はスタンプ結合解消と DFD レベリング深化であり、要件層 SPEC の細分化はそれと独立に判断できる。分析層改訂を先に確定し、要件層改訂を巻き込まないことで影響範囲を限定できる（PR7 設計フェーズ＝推奨で暫定前進）。(2) リーフ P→SPEC 必須辺（RULE-006）は傘 SPEC で満たせるため、傘マップ維持でも機械検証上の穴は生じない。(3) 細分化の要否は実装・テスト設計で「傘 SPEC では一意にテストできない」という具体的実害が出た時点で判断する方が、過剰な SPEC 新設（YAGNI）を避けられる。C は単一責務方針と逆行するため非推奨、B は要件層改訂が過大で本パスの目的外のため非推奨。

**G3（SPEC-29-1/29-2）への推奨適用**: 既存の推奨 A を SPEC-29-1/29-2 にも適用する。SPEC-29-1/29-2 の正常系アサーション（ゼロ件通過・価値経路到達充足）は当面 P-3-1 の傘 SPEC-29 マップで RULE-006 必須辺（P→SPEC）を満たし、テスト設計・実装で「傘 SPEC-29 では SPEC-29-1/29-2 を一意にテストできない」という実害が顕在化した時点で、対象リーフへ `→SPEC-29-1`/`→SPEC-29-2` の細分化マップを新設する。理由：(1) 傘 SPEC-29 マップで RULE-006 必須辺の機械検証上の穴は生じない、(2) 細分化マップ新設は実害が出た時点で判断する方が過剰 SPEC 化（YAGNI）を避けられる、(3) 本パスの主目的（スタンプ結合解消・DFD レベリング深化）と独立に要件層細分化を判断できる。

**ブロッカー**: 要件層 SPEC を細分化するか傘マップを維持するかは**オーナー判断**。実施時期（今スプリント細分化 vs 次スプリント以降）も含めてオーナーが決定する（**独断でのスプリント繰越は禁止**＝CLAUDE.md「スケジュール独断禁止」。本 Q は `scheduled` を空のままとし、判断を仰ぐ）。決定後は DD へ昇格（id 通貫）。
